# Codex + PPT Master：从资料到高质量 PPT 的工作流

这篇笔记整理一个实用流程：如何让 Codex 配合 PPT Master，把资料变成更稳定的课程课件、汇报材料或长文档演示稿。

适合读者：

- 经常把 PDF、Word、网页、Markdown 转成 PPT 的人。
- 想用 Agent 做课件，但不想得到一堆空泛页面的人。
- 想理解 PPT Master 和通用演示文稿工具如何分工的人。

## 一句话结论

高质量 PPT 不是“让 AI 直接生成 PPT”，而是把任务拆成四层：

```text
资料层：原始材料是否干净、可信、可引用。
策略层：这套 PPT 到底要说服谁、解决什么问题。
页面层：每一页只表达一个核心观点。
验收层：逐页检查事实、结构、视觉和可编辑性。
```

PPT Master 的价值在于把“资料到页面”的流程做得更结构化，而 Codex 的价值在于负责读资料、拆任务、调用工具、检查结果和迭代修改。

## 推荐工作流

### 1. 先整理资料，不要直接开做

输入可以是：

```text
PDF
DOCX
网页
Markdown
会议纪要
课程大纲
论文摘要和图注
```

开始前先让 Codex 做资料检查：

```text
1. 资料来源是否清楚。
2. 哪些内容可以直接引用。
3. 哪些数据需要核验。
4. 哪些段落适合做图表或流程图。
5. 哪些内容不适合进入 PPT。
```

如果资料本身混乱，直接生成 PPT 只会把混乱包装得更好看。

### 2. 先定八个决策

真正开始生成前，至少要确认：

```text
1. 受众是谁。
2. 目标是教学、汇报、路演还是复盘。
3. 页数范围。
4. 输出比例，例如 16:9。
5. 风格方向，例如专业、教学、学术、产品。
6. 是否需要讲稿。
7. 是否需要图表、流程图、架构图。
8. 哪些内容必须严格基于资料，不允许模型补编。
```

这一步决定 PPT 的上限。没有这些决策，后面很容易变成“看起来像 PPT，但不知道为什么存在”。

### 3. 先要大纲，再要页面

不要一开始就让 Agent 生成最终 PPT。

更稳的顺序是：

```text
1. 生成全局叙事线。
2. 生成每页标题。
3. 为每页写一句核心观点。
4. 标注每页证据来自哪段资料。
5. 标注每页适合的视觉形式。
6. 人工确认大纲。
7. 再进入页面生成。
```

每页最好只有一个核心观点。例如：

```text
差：Agent 工程化介绍
好：没有 Harness 的 Agent 很难复现线上错误
```

标题越像结论，PPT 越容易读。

### 4. PPT Master 和演示工具的分工

一个实用分工是：

```text
Codex：
  读资料
  拆结构
  写页面观点
  调用工具
  检查问题
  汇总修改意见

PPT Master：
  把资料和页面策略转成 SVG 页面
  管理页面风格
  处理多页一致性
  导出 PPTX

演示文稿工具：
  做最后格式检查
  处理人工微调
  用于最终交付或演讲
```

不要让任何一个工具独自承担全部工作。PPT 是“内容策略 + 页面设计 + 文件工程”的组合任务。

## 给 Codex 的可用指令

下面是一段适合启动工作的提示词：

```text
我想把这份资料做成 PPT。

资料：{粘贴资料或说明文件路径}
受众：{谁会看}
目标：{教学 / 汇报 / 路演 / 复盘}
页数：{大约页数}
风格：{专业 / 教学 / 学术 / 产品}

请先不要生成 PPT。
先完成以下工作：
1. 阅读资料并列出可用信息。
2. 标出需要事实核验的内容。
3. 给出 3 条可能的叙事线。
4. 推荐其中 1 条，并解释理由。
5. 生成每页标题、核心观点、证据来源和建议视觉形式。
6. 等我确认大纲后，再进入 PPT Master 生成流程。

约束：
- 不要编造资料里没有的数据。
- 每页只表达一个核心观点。
- 如果内容不够支撑某页，标注“需要补充资料”。
```

## 逐页验收清单

生成后不要只看整体好不好看，要逐页检查：

```text
1. 这一页有没有明确观点？
2. 观点是否来自资料，还是模型自己补的？
3. 标题是不是结论句？
4. 页面信息是否过密？
5. 图表和文字是否表达同一件事？
6. 字号、对齐、留白是否稳定？
7. 是否有无法编辑或导出异常的元素？
8. 讲稿是否和页面同步？
```

PPT 的失败通常不是一页突然坏掉，而是每页都“差一点”：标题不够结论化、要点太多、图表没有服务观点、讲稿和页面脱节。

## 常见错误

### 错误 1：把“生成 PPT”当成单步任务

更好的做法是拆成：

```text
资料清洗 -> 叙事设计 -> 页级大纲 -> 页面生成 -> 逐页验收 -> 导出修订
```

### 错误 2：没有确认受众

同一份资料给老板、学生、客户、工程师看，页面结构完全不同。

### 错误 3：让模型补数据

如果资料没有数据，就让它标注缺口，不要让它补一个“看起来合理”的数字。

### 错误 4：页面标题只写名词

标题最好写成观点句，不要只写分类名。

```text
差：风险控制
好：高风险工具调用必须进入人工审批
```

### 错误 5：只验收视觉，不验收可编辑性

最终交付的是 PPTX，不是截图。需要检查导出文件能否打开、文字是否可读、页面是否错位、讲稿是否保留。

## 什么时候不适合用这套流程

以下场景不建议上重流程：

```text
只要 3 页以内的临时介绍。
内容还没想清楚，只是在头脑风暴。
资料版权不清楚，不能用于公开演示。
需要完全遵循公司模板但模板文件没有提供。
```

## 最终建议

用 Codex + PPT Master 做 PPT，核心不是追求“一句话出片”，而是追求稳定：

```text
资料可追溯。
结构可讨论。
页面可验收。
文件可交付。
后续可修改。
```

这才是 Agent 做 PPT 真正有价值的地方。
