# Codex Goal 指令怎么写：把模糊需求变成可执行目标

## 先说结论

很多 Agent 任务失败，不是因为模型完全不会做，而是因为用户一开始给的目标太松。

例如：

```text
帮我做个好看的页面。
帮我优化一下仓库。
帮我把这个功能做完整。
```

这些话对人类有语感，但对 Agent 来说缺少 6 件事：

1. 做到什么算完成。
2. 怎么验证完成。
3. 哪些地方不能碰。
4. 能改哪些文件或系统。
5. 失败后最多迭代几轮。
6. 遇到什么情况必须暂停让人确认。

`/goal` 的价值，就是把一句模糊愿望改写成一份可执行、可验证、有边界的任务合同。

## 一个好 Goal 的 7 个字段

| 字段 | 要回答的问题 | 常见错误 |
|---|---|---|
| 目标结果 | 用户最后能看到或拿到什么？ | 只写“优化”“高级”“完整” |
| 验证证据 | 用什么命令、截图、文件、链接或流程证明完成？ | 只写“确认可用” |
| 约束 | 哪些能力、素材、数据、账号或范围不能使用？ | 默认允许 Agent 自己扩范围 |
| 写入边界 | 允许改哪些目录、文件、配置或发布目标？ | 写成“随便改” |
| 迭代策略 | 先做哪一版，最多改几轮，失败后看什么证据？ | 无限重试或没有顺序 |
| 完成条件 | 什么同时满足后可以收尾？ | 只跑通一部分就结束 |
| 暂停条件 | 哪些情况必须停下来问人？ | 账号、支付、生产、版权、隐私都没写 |

这 7 个字段不一定每次都要写成完整长文。小任务可以压缩成 5-10 行，但不能缺少验证和边界。

## 从一句话到 Goal 的改写流程

| 步骤 | 要做什么 | 示例 |
|---|---|---|
| 1. 找核心产物 | 把“优化仓库”改成“新增一篇可公开笔记并接入目录” | 产物是 `.md` 文件和索引链接 |
| 2. 写最小验证 | 说明怎么确认它不是空内容 | Markdown 链接检查、隐私扫描、git diff 审计 |
| 3. 收窄边界 | 限定只改相关文件 | 只改 `notes/`、`README.md`、目录索引 |
| 4. 标出不做 | 防止 Agent 自己加功能 | 不创建营销页、不上传私有材料、不改无关配置 |
| 5. 设暂停条件 | 把高风险动作从自动执行里拿掉 | 需要账号发布、生产部署、客户数据时暂停 |

## 可复制模板

```text
/goal [用一句话写清最终产物和用户可见结果]

验证：
- [命令、截图、链接、文件、浏览器流程或人工检查标准]
- [失败时需要保留的日志或证据]

约束：
- 不处理未脱敏隐私、账号凭据、付款、生产数据或未授权素材。
- 不做和目标无关的重构、发布或范围扩张。

边界：
- 只读取/修改 [允许的目录、文件、页面或系统]。
- 如果发现必须触碰边界外内容，先停下来说明原因。

迭代策略：
- 先完成最小可用版本，再做最多 [N] 轮聚焦改进。
- 每轮改动后根据 [测试、截图、链接检查、用户流程] 验证。

完成条件：
- [核心流程或核心产物] 已完成。
- [验证证据] 已通过或明确列出无法通过的原因。
- 没有新增隐私、版权、账号或生产风险。

暂停条件：
- 需要真实账号登录、付款、生产发布、读取密钥、处理未脱敏数据、使用未授权素材，或目标范围发生重大变化。
```

## 四个常见场景

### 1. 做一个小网页或工具

不要只写：

```text
/goal 做一个好看的 Agent 学习网站。
```

更稳的写法：

```text
/goal 做一个本地可运行的 Agent 新手学习页，首页直接展示 7 天路线、3 个练习入口和 1 个资料导读入口。

验证：启动本地服务，在桌面和移动视口各截图一次；确认首屏没有文字重叠，所有入口链接可点击。
约束：不做登录、支付、后端、真实发布和营销页。
边界：只改当前项目的页面、样式和必要的本地测试文件。
完成条件：本地页面可访问，三类入口都能打开，移动端无明显布局破损。
暂停条件：需要上线、接账号、接数据库或使用第三方版权素材时暂停。
```

### 2. 修 bug

```text
/goal 修复 [具体 bug]，让 [用户动作] 在 [环境] 下得到 [正确结果]。

验证：先复现 bug，记录复现步骤；修复后运行相关测试；如果是 UI bug，用浏览器走通一次对应流程并截图。
边界：优先改和 bug 直接相关的模块，不做无关重构。
暂停条件：复现不了、需要生产数据、需要改公共接口或会影响兼容性时暂停说明。
```

### 3. 整理公开资料

```text
/goal 把 [公开链接或公开主题] 整理成一篇可发布 Markdown 笔记，读者能在 10 分钟内理解核心概念、适用场景和下一步练习。

验证：保留来源链接；不复制长篇原文；完成后检查 Markdown 内链和隐私风险。
边界：只使用公开网页、官方文档或用户明确提供的材料。
暂停条件：遇到登录墙、付费墙、版权限制或无法确认来源时暂停。
```

### 4. 维护一个公开仓库

```text
/goal 维护公开知识库的一次小更新：新增或改进一篇已审计内容，并把它接入 README、目录页或对应专题页。

验证：运行 Markdown 链接检查、隐私关键词扫描和 git diff 审计；推送后确认远端文件可访问。
约束：不上传本地私有 skill 源码、真实聊天记录、账号后台、密钥、个人路径或未脱敏截图。
边界：只改和本次内容相关的 Markdown 文件。
完成条件：本地检查通过，远端仓库更新成功，变更摘要清楚。
暂停条件：需要真实账号操作、生产发布、处理隐私材料或 GitHub 权限失败时暂停说明。
```

## 判断 Goal 好不好的检查表

写完后问自己：

| 问题 | 如果答案是否，怎么补 |
|---|---|
| 读者能看出最终产物吗？ | 把抽象动词改成文件、页面、功能、链接、截图或报告 |
| Agent 知道怎么验证吗？ | 写命令、流程、截图、检查表或人工审核点 |
| Agent 知道不能做什么吗？ | 写不做登录、不做支付、不读密钥、不碰生产、不用版权素材 |
| Agent 知道能改哪里吗？ | 写目录、模块、页面或系统边界 |
| Agent 知道什么时候停吗？ | 写暂停条件和完成条件 |

## 和 qiaomu-goal-meta-skill 的关系

`qiaomu-goal-meta-skill` 已经把这套思路做成可安装 skill。它适合在你只有一句模糊需求时，先生成一版推荐 `/goal`，再让你用少量选项收窄范围。

这篇笔记不复制它的原始长提示词，只提炼新手最应该学会的结构：目标、验证、约束、边界、迭代、完成、暂停。

## 公开分享边界

- 可以分享：写 Goal 的字段、示例、检查表和公开仓库链接。
- 不要分享：真实项目路径、账号、客户材料、未脱敏截图、内部发布流程、密钥或私有提示词全文。
- 涉及医疗、金融、法律、版权、生产数据、支付或账号操作时，Goal 的默认策略应该是“发现优先 + 人工确认”，不要直接自动执行。
