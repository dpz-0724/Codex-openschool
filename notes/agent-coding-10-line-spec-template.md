# Agent 编程前的 10 行 Spec 模板：先把需求写成可验收任务

## 审计信息

- 整理时间：2026-06-12。
- 来源：AIHot 最近 7 天精选里的 Spec / PRD / Agent Coding 条目；结合 `warpdotdev/common-skills`、GitHub Spec Kit、`qiaomu-ai-prd` 公开仓库和本地 Codex 任务执行经验整理。
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 适合读者：已经会让 Agent 写代码、做页面或改文件，但经常遇到“做出来能跑，却不是我要的”的新手。
- 目标：给新手一个 10 行以内就能复制使用的 Spec 模板，在让 Agent 动手前先写清目标、边界和验收。
- 隐私处理：不包含本地路径、账号、客户材料、私有仓库、真实业务数据、密钥或未发布项目。

## 先说结论

Agent 编程最常见的问题不是模型不会写代码，而是用户把“想法”直接丢给 Agent，让它边猜边实现。

Spec 的作用就是把一句话需求变成可验收任务：

```text
想法：帮我做一个知识库页面。
Spec：谁用、解决什么问题、必须有什么、不做什么、怎么验收。
```

对新手来说，不需要一上来就安装完整 SDD 工具。先学会写 10 行 Spec，就能明显减少返工。

## 10 行 Spec 模板

把下面这段复制给 Agent，先让它补全，不要直接开工：

```text
任务名称：
目标用户：
当前问题：
期望行为：
必须满足的不变量：
明确不做：
输入材料：
输出产物：
边界情况：
验收方式：
```

每一行只写一两句话。写不出来的地方，不要让 Agent 猜，标成“待确认”。

## 每一行怎么写

| 行 | 写什么 | 常见错误 |
|---|---|---|
| 任务名称 | 一个短标题，说明这次要完成的事 | 写成“优化一下”“弄好看点” |
| 目标用户 | 谁会使用或阅读这个产物 | 只写“用户”，没有角色 |
| 当前问题 | 现在哪里不好，为什么要做 | 跳过问题，直接给方案 |
| 期望行为 | 用户做什么，系统或文档怎么响应 | 写实现细节，不写可见行为 |
| 必须满足的不变量 | 无论怎么实现都不能破坏的规则 | 写成口号，例如“体验好” |
| 明确不做 | 本次不包含什么，避免范围膨胀 | 什么都想做 |
| 输入材料 | URL、文件、截图、数据、仓库、上下文 | 不说明来源和版本 |
| 输出产物 | Markdown、代码、PPT、页面、报告、截图 | 只说“完成” |
| 边界情况 | 空数据、失败、权限、移动端、长文本、异常输入 | 默认只考虑 happy path |
| 验收方式 | 命令、截图、链接、清单、人工检查点 | 写“你自己检查一下” |

## 一个可直接用的例子

假设你要让 Agent 给个人知识库新增一篇教程，不要这样说：

```text
帮我写一篇 Agent 笔记，顺便放到仓库里。
```

先写成：

```text
任务名称：新增 Agent 工具选择入门笔记。
目标用户：刚知道 Agent 可以调用工具，但分不清 API、CLI、MCP、浏览器自动化的新手。
当前问题：新手容易一上来追热门协议，却没有先判断任务需要读数据、执行动作还是编排流程。
期望行为：写一篇 Markdown 笔记，用表格和判断流程解释不同工具形态的适用场景。
必须满足的不变量：只使用公开资料；每个外部观点保留来源；不出现本地路径、账号、密钥或私有项目。
明确不做：不做 MCP 教程全集；不推荐未验证工具；不上传本地私有 skill 源码。
输入材料：仓库已有 notes/、skills/、prompts/ 目录，以及公开官方文档链接。
输出产物：一篇可直接阅读的 notes/*.md，并更新 README 和 notes/README.md。
边界情况：如果发现已有笔记重复，先合并或补充，不新增低质量重复文章。
验收方式：Markdown 内部链接全通；敏感扫描无命中；README 能从首页点到新笔记；远程 GitHub 可读取。
```

这样 Agent 执行时会更稳，因为它知道：

- 读者是谁。
- 要解决什么问题。
- 哪些内容不能碰。
- 最后用什么证明完成。

## Product Spec、Tech Spec、Verify 怎么分

AIHot 里的 Spec 驱动开发条目提到 `Spec -> Implement -> Verify` 闭环。新手可以先简化成三张纸：

| 文档 | 解决什么 | 适合什么时候写 |
|---|---|---|
| Product Spec | 用户看见什么、能做什么、哪些行为必须成立 | 有 UI、流程、内容结构、业务规则时 |
| Tech Spec | 准备怎么实现、会改哪些模块、风险在哪里 | 涉及多文件、多模块、架构取舍时 |
| Verify | 怎么证明实现真的符合前两份 Spec | 每次都要有，哪怕只是检查清单 |

最小任务可以只写 10 行 Spec。中等任务写 `PRODUCT.md` 加验收清单。大任务再补 `TECH.md`、任务拆分和实现计划。

## 什么时候不需要 Spec

Spec 不是仪式。下面这些任务通常不需要专门写文档：

- 改一个错别字。
- 调整一行链接。
- 修一个已经定位清楚的单行 bug。
- 运行一次格式化。
- 回答一个概念问题，不涉及文件变更。

判断标准很简单：如果 Agent 猜错代价很小，直接做；如果猜错会返工、破坏文件、影响用户或引入隐私风险，先写 Spec。

## 让 Agent 先问问题，而不是先写代码

你可以这样提示：

```text
先不要实现。
请把我的需求改写成 10 行 Spec。
如果有不确定项，用“待确认”标出来。
不要自行补全会影响范围、权限、数据或验收方式的内容。
我确认 Spec 后，你再开始实现。
```

这条提示的关键不是“写得正式”，而是阻止 Agent 过早进入实现。

## 实现时怎么用 Spec

确认 Spec 后，再给 Agent 一条执行提示：

```text
请严格按上面的 10 行 Spec 执行。
实现过程中不得扩大“明确不做”的范围。
如果发现 Spec 与现有代码或资料冲突，先停下来说明冲突。
完成后按“验收方式”逐条给出证据。
```

如果任务涉及代码，再补两行：

```text
先列出会修改的文件和原因，再编辑。
完成后运行相关测试或说明为什么无法运行。
```

## Verify：不要只问“做完了吗”

验收要能对照 Spec，一条一条看证据。

| 验收项 | 好证据 | 弱证据 |
|---|---|---|
| 文件已生成 | 文件路径、关键标题、目录入口 | Agent 说“已写好” |
| 代码能跑 | 测试命令和结果 | “看起来没问题” |
| 页面正确 | 截图、浏览器检查、控制台无错 | 只读代码不看页面 |
| 来源可信 | 一手链接、引用位置 | 没有来源的摘要 |
| 没有隐私泄露 | 敏感扫描命令无命中 | 人眼大概看过 |
| 和 Spec 一致 | 按 10 行逐条对照 | 泛泛总结 |

如果某一条没有证据，就不要说完成。

## 常见失败模式

### 1. Spec 太像愿望

```text
目标：做一个高级、好用、现代化的页面。
```

改成：

```text
目标：做一个新手打开后 30 秒内能找到学习路线、Skills 清单和提示词入口的仓库首页。
```

### 2. 不变量写成了实现方式

```text
必须使用 React 和 Tailwind。
```

如果用户真正关心的是页面效果，应改成：

```text
必须在桌面和手机上都能清楚展示目录，按钮文字不换行溢出，首屏能看到下一段内容。
```

技术栈可以放 Tech Spec，不要混进 Product Spec。

### 3. 明确不做为空

没有“不做什么”，Agent 很容易越界。例如：

```text
明确不做：不接真实支付；不读取私有数据库；不自动发布到生产；不新增未验证第三方依赖。
```

### 4. 验收方式不可执行

```text
验收方式：效果要好。
```

改成：

```text
验收方式：运行 npm test；用浏览器打开本地页面；检查移动端 390px 和桌面 1440px 截图；确认无控制台错误。
```

## 和完整 SDD 工具的关系

这篇笔记不是让新手立刻上完整工程体系。

可以按任务规模分层：

| 任务规模 | 推荐做法 |
|---|---|
| 30 分钟以内的小任务 | 10 行 Spec + 验收清单 |
| 1-2 天的功能 | `PRODUCT.md` + 简短实现计划 + Verify |
| 多模块功能 | `PRODUCT.md` + `TECH.md` + 任务拆分 |
| 团队协作或长期项目 | 引入 Spec Kit、shared skills、PR 审查和 trace |

完整工具的价值在于把 spec、plan、tasks、implement 和 analyze 变成结构化流程；但新手第一步应该先学会在任何 Agent 里写清楚“做什么、不做什么、怎么验收”。

## 可复制提示词

### 把想法改写成 Spec

```text
你是我的 Agent 任务规格审计员。
先不要实现，也不要写代码。
请把下面的需求改写成 10 行 Spec：

任务名称：
目标用户：
当前问题：
期望行为：
必须满足的不变量：
明确不做：
输入材料：
输出产物：
边界情况：
验收方式：

要求：
1. 每行只写 1-2 句。
2. 不确定的地方写“待确认”，不要自行脑补。
3. 标出哪些问题必须在实现前问我。
4. 最后给出“可以开始实现 / 不建议开始实现”的判断。

原始需求：
[粘贴你的需求]
```

### 按 Spec 执行并验收

```text
请按下面的 10 行 Spec 执行任务。

规则：
1. 先复述你理解的目标和不做范围。
2. 列出计划修改的文件、工具和验证方式。
3. 不得扩大范围；发现冲突先停下来说明。
4. 完成后按“验收方式”逐条给出证据。
5. 如果某条无法验证，明确写“未验证”，不要说已完成。

Spec：
[粘贴 10 行 Spec]
```

### 审计实现是否偏离 Spec

```text
请审计当前实现是否符合下面的 Spec。

输出格式：
| Spec 条目 | 证据 | 结论 | 需要修复吗 |

判断规则：
- 只有看到文件、命令输出、截图、链接或测试结果，才算证据。
- 意图、猜测和“看起来可以”不算证据。
- 如果实现扩大了“明确不做”的范围，标为高风险。

Spec：
[粘贴 Spec]
```
