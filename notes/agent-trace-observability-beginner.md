# Agent 运行日志与 Trace：别只看最终回答

## 审计信息

- 整理时间：2026-06-12。
- 来源：OpenAI Agents SDK Tracing 文档、OpenAI Cookbook 的 Agent improvement loop 与 macro evals 示例、Claude connector observability 博客、Claude Code monitoring 文档，以及本仓库已有长任务和代码审查笔记。
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 适合读者：已经开始让 Agent 查网页、改文件、跑命令或操作浏览器，但还不知道怎么验收结果的新手。
- 目标：给新手一套最小证据包模板，避免把 Agent 的“我完成了”当成完成。
- 隐私处理：不包含本地路径、账号、聊天记录、客户材料、浏览器历史、私有仓库、密钥、后台截图或内部运营目标。

## 先说结论

Agent 的最终回答不是证据。

如果一个 Agent 说：

```text
我已经整理好了，链接也检查过了。
```

你至少要能看到：

```text
它读了哪些来源？
它调用了哪些工具？
它改了哪些文件？
它运行了哪些检查？
哪些地方失败过？
哪些结论只是推断？
哪些动作需要人确认？
```

这组东西可以叫运行日志、trace、证据包、复盘记录。名字不重要，重要的是：任务完成后，你能回放它到底做了什么。

## 为什么新手最容易忽略 Trace

新手通常盯着最终答案看：

- 文案是不是顺。
- 表格是不是完整。
- 结论看起来是不是合理。

但 Agent 真正容易出问题的地方通常藏在过程里：

| 问题 | 只看最终回答会漏掉什么 |
|---|---|
| 网页调研 | 可能没有读一手来源，只是引用搜索摘要 |
| 代码修改 | 可能改了文件但没有跑测试，或测试失败被一句话带过 |
| 浏览器自动化 | 可能点错页面、读到登录态数据、截图含隐私 |
| 长任务 | 可能中途换了目标，最后只交一个漂亮总结 |
| 内容发布 | 可能把内部材料、账号信息或未脱敏截图带进公开仓库 |

Trace 的价值不是让流程看起来专业，而是让你能判断：这个结果是不是可信。

## Trace、日志、Artifact、报告有什么区别

| 名称 | 新手可以怎么理解 | 例子 | 验收重点 |
|---|---|---|---|
| Trace | 一次任务从开始到结束的结构化过程记录 | 模型调用、工具调用、handoff、guardrail、custom span | 能不能回放关键动作 |
| 日志 | 按时间记录发生了什么 | 命令输出、错误、重试、检查结果 | 有没有失败和异常 |
| Artifact | 任务真正产生的文件或证据 | Markdown、diff、截图、表格、报告、测试结果 | 能不能打开和复查 |
| 最终报告 | 给人读的总结 | 本次完成什么、风险是什么、下一步是什么 | 是否引用证据而不是空口判断 |

OpenAI Agents SDK 文档把 trace 描述成一次 Agent run 中事件的完整记录，包含 LLM 生成、工具调用、handoff、guardrail 和自定义事件；它还把 trace 拆成 spans，用来表达具体操作的开始和结束。对新手来说，不需要一开始就搭完整观测系统，但要学会保留同样类型的证据。

## 最小证据包：每次 Agent 任务都留这 8 项

| # | 项目 | 要回答的问题 | 最小写法 |
|---:|---|---|---|
| 1 | 输入 | 用户到底让 Agent 做什么 | 原始需求或简短复述 |
| 2 | 边界 | 哪些事允许做，哪些不允许 | 只读范围、可写范围、禁止动作 |
| 3 | 来源 | 事实从哪里来 | URL、文件名、日期、版本 |
| 4 | 工具 | Agent 调了什么工具 | 搜索、浏览器、shell、git、测试 |
| 5 | 产物 | 结果在哪里 | 新增文件、修改文件、截图、表格 |
| 6 | 检查 | 怎么证明结果可用 | 测试、链接检查、隐私扫描、人工复核 |
| 7 | 失败 | 哪些尝试没成功 | 404、权限不足、测试失败、来源不可核 |
| 8 | 人工确认 | 哪些动作需要人批准 | 发布、删除、提交、部署、读取敏感资料 |

如果只能保留一句话，就保留这句：

```text
完成结果 = 产物 + 来源 + 检查结果 + 已知风险。
```

## 四类任务怎么留证据

### 1. 网页调研

适合任务：

- 整理一篇公开网页。
- 调研一个开源项目。
- 从官方文档提炼教程。
- 把 AIHot 条目拆成学习笔记。

最小证据包：

| 证据 | 示例 |
|---|---|
| 来源 URL | 官方博客、GitHub README、论文页、产品文档 |
| 访问日期 | 2026-06-12 |
| 核心结论 | 只写能被来源支持的结论 |
| 未确认项 | 只看到社交摘要、没有官方文档、项目未实测 |
| 版权处理 | 不复制长篇原文，只转述和短引 |
| 隐私处理 | 不放账号、聊天、后台、路径、截图里的个人信息 |

可复制提示词：

```text
请整理这组公开来源，但不要只给最终总结。

输出：
1. 你读取的来源 URL 和访问日期。
2. 每个关键结论对应的来源。
3. 哪些结论是一手来源支持，哪些只是推断。
4. 是否存在过度引用、隐私、账号、后台截图或版权风险。
5. 最终 Markdown 笔记。
6. 你建议我人工复核的 3 个点。
```

### 2. 代码修改

适合任务：

- 让 Codex 改一个 bug。
- 让 Agent 增加一个页面。
- 更新 README、目录和链接。
- 重构一段脚本。

最小证据包：

| 证据 | 示例 |
|---|---|
| 文件范围 | `notes/*.md`、`README.md` |
| diff | 本次实际改动 |
| 检查命令 | `git diff --check`、测试命令、lint |
| 失败输出 | 哪些测试失败，是否和本次改动有关 |
| 回滚边界 | 不使用 destructive git 命令，不覆盖用户改动 |
| 人工确认 | 删除文件、提交远端、发布版本前确认 |

可复制提示词：

```text
请完成这个代码或文档修改，并在结束时给我证据包。

必须包含：
1. 改了哪些文件，为什么改。
2. 关键 diff 的行为影响。
3. 运行了哪些验证命令，结果是什么。
4. 哪些验证没有运行，原因是什么。
5. 是否发现无关的现有改动。
6. 是否需要我确认提交、发布、删除或部署。
```

### 3. 浏览器自动化

适合任务：

- 操作公开网页。
- 调试本地页面。
- 截图验证 UI。
- 用当前浏览器登录态读取页面。

最小证据包：

| 证据 | 示例 |
|---|---|
| 目标页面 | URL、页面标题 |
| 操作路径 | 打开、点击、输入、截图、读取 DOM |
| 视觉证据 | 截图或关键 DOM 文本 |
| 登录边界 | 是否使用了真实账号或当前浏览器状态 |
| 隐私检查 | 截图是否包含头像、邮箱、订单、聊天、后台数据 |
| 停止条件 | 遇到验证码、登录墙、付款、发布、删除就停 |

可复制提示词：

```text
请操作这个网页，但每一步都要保留可复查证据。

输出：
1. 打开的 URL 和页面标题。
2. 实际点击或输入了什么。
3. 关键截图或 DOM 摘要。
4. 是否使用登录态，是否看到隐私数据。
5. 哪些动作没有继续做，原因是什么。
6. 最终状态是否达到目标。
```

### 4. 长任务

适合任务：

- 连续维护一个知识库。
- 跑 30 分钟以上的资料整理。
- 让 Agent 自主搜索、筛选、写作、更新目录。
- 做多文件、多来源、多轮验证。

最小证据包：

| 证据 | 示例 |
|---|---|
| 计划 | 本轮做什么，不做什么 |
| checkpoint | 每个阶段已完成、证据、风险 |
| 预算 | 时间、来源数量、命令次数、工具调用 |
| 停止条件 | 需要登录、来源不可核、涉及隐私、范围扩大 |
| 最终产物 | 文件、链接、检查记录 |
| 下一步 | 继续做什么，先不做什么 |

可复制提示词：

```text
请把这个长任务当成可审计任务执行。

执行前先输出：
1. 计划。
2. 可读范围。
3. 可写范围。
4. 禁止动作。
5. 预算和停止条件。

执行后输出：
1. 实际步骤。
2. 产物列表。
3. 来源和检查结果。
4. 失败或未完成项。
5. 需要人工确认的动作。
```

## 什么内容不要公开进仓库

证据包不是把所有日志原样公开。

公开仓库里不要放：

- 本地绝对路径、用户名、浏览器 profile、机器名。
- API 凭证、访问令牌、cookie、`.env`、授权 header。
- 真实聊天记录、客户材料、订单、联系人、账号后台。
- 未脱敏截图，尤其是邮箱、头像、余额、工单、私信。
- 内部运营目标、增长计划、发布节奏、工作手册。
- 失败过程里包含的敏感命令或完整错误堆栈。
- 版权材料的长篇原文。

推荐做法：

| 原始材料 | 公开写法 |
|---|---|
| 本地路径 | 写成“本地笔记目录”或不写 |
| 私有聊天 | 提炼成“用户反馈类型” |
| 后台截图 | 改成文字结构或脱敏示意 |
| 内部目标 | 不进公开仓库 |
| 命令输出 | 只保留必要结论和错误类型 |
| 长篇原文 | 只写摘要、链接和短引用 |

## 用 Trace 反过来提升 Agent

OpenAI Cookbook 的 Agent improvement loop 示例给了一个很重要的思路：不是把 trace 看完就结束，而是把真实 trace、人工反馈、模型反馈和 eval 组合起来，变成下一轮可执行的改进。

新手可以先做一个轻量版本：

```text
1. 跑一次任务。
2. 留下证据包。
3. 标出哪里做得好、哪里没有证据、哪里超出边界。
4. 把问题改成下一次提示词里的规则。
5. 用同类任务再跑一次，看问题是否减少。
```

例子：

| 发现的问题 | 下一轮规则 |
|---|---|
| Agent 引用了搜索摘要 | 每个关键事实必须补一手来源 |
| Agent 说测试通过但没给命令 | 最终报告必须列出验证命令和结果 |
| Agent 截图里有账号信息 | 浏览器自动化前先声明隐私边界，截图前检查页面 |
| Agent 长任务跑偏 | 开始前写停止条件和 checkpoint |
| Agent 改了无关文件 | 先写可写范围，结束时列出改动文件 |

这就是把一次失败变成下一次流程质量的方式。
