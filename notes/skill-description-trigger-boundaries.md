# Skill 描述怎么写才不误触发：description 是触发器，不是宣传语

## 审计信息

- 整理时间：2026-06-12。
- 来源：AIHot 最近 7 天精选里的 Skills / Agent 条目；结合 Replit Agent Skills 官方博客与文档、本地 Codex skill-creator 原则、本仓库已有 skills 审计经验整理。
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 适合读者：想安装、创建或审核 Agent skills，但经常分不清“什么值得做成 skill”的新手。
- 目标：教会读者把 skill 的 `description` 写成触发边界，而不是写成宣传语。
- 隐私处理：不包含本地路径、账号、聊天记录、私有 skill 源码、密钥或未发布项目。

## 先说结论

一个好 skill 不是“我很强，可以帮你做很多事”，而是：

> 当用户提出某类可重复任务时，Agent 能通过 `description` 判断该不该加载这份操作手册。

所以 `description` 不是简介，也不是卖点文案，而是触发器。它至少要回答四件事：

1. 这个 skill 解决什么具体问题？
2. 用户说什么、提供什么材料时应该触发？
3. 哪些场景不应该触发？
4. 触发后应该产出什么，如何验收？

如果这四件事说不清，通常说明它还不该做成公开 skill。

## 为什么这件事突然重要

AIHot 近期有一批 Skills 相关条目：Replit Agent 新增 Custom Instructions 和 Skills，Spec 驱动开发用三段 Skills 固化 `Spec -> Implement -> Verify` 闭环，qiaomu-ai-prd 把 PRD 改造成更适合 Agent 执行的输入，Text-To-Lottie 把 skill 和本地预览 harness 组合起来做可验收动画，Teach skill 把工作目录变成有状态学习空间。

这些条目的共同点不是“大家都在发 skill”，而是 Agent 工作流正在从临时提示词变成可复用的任务协议。

但一旦 skill 多起来，真正的问题会变成：

- 什么时候触发这个 skill？
- 它和已有 skill 是否重复？
- 它会不会抢走不该处理的任务？
- 它有没有把隐私、权限、发布动作写清楚？

Replit 文档里也强调了同一件事：Agent 会先看 skill 的名称和描述，只有相关时才加载完整内容。也就是说，`description` 写错，后面写再多正文也救不了触发错误。

## Custom Instructions 和 Skills 的边界

先把两个概念分开：

| 类型 | 什么时候用 | 例子 | 风险 |
|---|---|---|---|
| Custom Instructions | 每次对话都应该遵守的长期偏好或团队约定 | 默认语言、代码风格、提交规范、不要上传隐私 | 写太多会污染所有任务 |
| Skill | 某类任务出现时才需要加载的操作手册 | 网页整理成 Markdown、PPT 生成、Agent 自动化浏览、Spec 审计 | 触发边界太宽会误触发 |

判断口诀：

- 总是要遵守的规则，放 Custom Instructions。
- 只有特定任务才需要的流程，做成 Skill。
- 只是一次性的提示词，不要急着做成 Skill。
- 需要真实账号、私有数据或高风险操作才能讲清楚的，不要公开。

## 发布前先填这张表

写 skill 前先不要写正文，先把这张表填出来：

| 问题 | 必须写清的内容 |
|---|---|
| 解决什么问题 | 一句话说明任务，不写“提升效率”这种空话 |
| 触发场景 | 用户怎么说、会给什么输入、上下文是什么 |
| 输入 | URL、文件、截图、仓库、表格、主题、数据范围 |
| 输出 | Markdown、PPT、报告、代码、截图、审计清单 |
| 不适用场景 | 哪些任务应该交给别的 skill 或普通回答 |
| 风险动作 | 登录、发布、付款、删除、提交代码、读取私密数据 |
| 验收方式 | 命令、截图、链接、diff、lint、人工审核点 |

填不满这张表时，先把它当成提示词模板或笔记，不要急着发布成 skill。

## description 的写法

可以用这个结构：

```yaml
description: >
  [这个 skill 做什么]. Use when [具体触发场景、用户输入、任务上下文].
  Do not use for [不适用场景、重复能力、风险边界].
  Inputs: [主要输入].
  Outputs: [主要输出和验收材料].
  Requires confirmation before [高风险动作].
```

中文仓库也可以写中文，但逻辑一样：

```yaml
description: >
  用于把公开网页整理成可入库的 Markdown 学习笔记。
  当用户提供公开 URL，并要求摘要、拆解、学习笔记、资料入库时使用。
  不用于登录态网页、付费内容、批量抓取、复制全文或无版权授权的搬运。
  输入是 URL 和笔记目标，输出是带来源链接、结构化标题和自检清单的 Markdown。
  遇到登录、验证码、下载文件或发布内容前必须先确认。
```

这段描述不是为了好看，而是为了让 Agent 在任务开始前做判断。

## 常见坏写法

### 坏写法 1：范围太大

```yaml
description: 帮我处理网页相关任务。
```

问题：

- “网页相关”太宽，搜索、抓取、登录、测试、UI 验收都会抢着触发。
- 没写输入输出。
- 没写不适用场景。

改法：

```yaml
description: >
  用于从公开网页提取正文、标题、作者、时间和来源链接，并整理成 Markdown 摘要。
  当用户提供公开 URL，并要求网页笔记、资料整理、内容入库时使用。
  不用于登录态页面、需要交互操作的网页、批量爬取或 UI 自动化测试。
  输出必须包含来源链接、摘要、要点和可复查引用。
```

### 坏写法 2：只写工具名

```yaml
description: PPT 助手。
```

问题：

- 不知道是做大纲、设计、生成 PPTX、审校版式，还是转格式。
- 很容易和通用文档、图片、演示文稿能力重叠。

改法：

```yaml
description: >
  用于把论文、网页、Markdown 或 Word 文档整理成可演示的 PPT 方案。
  当用户要求课程 PPT、汇报 deck、长文档转演示、PPT 大纲和页面结构时使用。
  不用于单张海报、普通文档排版或未授权资料的全文搬运。
  输出应包含页级大纲、视觉风格、关键图表建议和交付验收点。
```

### 坏写法 3：把愿望写成能力

```yaml
description: 帮助 Agent 自我进化，越来越聪明。
```

问题：

- “自我进化”听起来高级，但触发边界不清楚。
- Agent 不知道什么时候应该用，也不知道完成标准。

改法：

```yaml
description: >
  用于复盘一次 Agent 任务，并把重复流程沉淀成提示词、笔记或候选 skill。
  当用户要求总结工作流、提炼可复用步骤、分析失败原因、更新知识库时使用。
  不用于实时执行高风险操作、读取私密历史记录或自动修改未审计的全局配置。
  输出应包含任务复盘、可复用流程、风险点和下一步验证方式。
```

## 判断一个内容是否值得做成 skill

不是所有好提示词都值得做成 skill。公开前可以用这 7 条筛：

| 判断项 | 通过标准 | 不通过时怎么处理 |
|---|---|---|
| 重复性 | 这个任务会反复出现 | 只做一次性提示词 |
| 输入稳定 | 输入形态可以说清楚 | 先写成教程，不做 skill |
| 输出稳定 | 产物有固定格式或验收点 | 先补输出模板 |
| 边界清楚 | 能写出不用它的场景 | 缩小范围 |
| 不重复 | 和已有 skill 分工明确 | 合并或只保留主入口 |
| 可验证 | 能通过命令、截图、链接或人工审查验收 | 先补验证流程 |
| 可公开 | 不依赖隐私材料才能说明价值 | 只发脱敏方法，不发源码 |

## 误触发诊断表

| 现象 | 可能原因 | 修法 |
|---|---|---|
| Skill 什么任务都触发 | `description` 太宽，只有能力描述，没有场景 | 加具体触发词和不适用场景 |
| Skill 该触发时没触发 | 描述里没写用户真实会说的话 | 补自然语言触发短语和输入形态 |
| 触发后输出很空 | 正文只有原则，没有流程、模板或验收标准 | 补步骤、产物格式和检查清单 |
| 和别的 skill 抢任务 | 名称和 description 范围重叠 | 合并、改名或明确优先级 |
| 越做越不稳定 | skill 依赖外部服务、模型或项目结构变化 | 加版本、日期和维护规则 |
| 公开后有隐私风险 | description 或示例暴露路径、账号、客户数据 | 脱敏重写，只保留方法 |

## 从 AIHot 近期条目提炼出的 4 个趋势

### 1. Skill 会越来越像任务协议

Spec 驱动开发 Skills 的重点不是“多三个命令”，而是把需求、实现和验证拆开，让 Agent 每一步都有可审查材料。

新手可以学习它的结构：

```text
目标：要完成什么。
不变量：必须始终成立的行为。
实现约束：不能破坏什么。
验证方式：怎么证明做对了。
```

### 2. Skill 需要和本地验证环境绑定

Text-To-Lottie 这类条目的价值不只是“能生成动画”，而是 skill 后面有本地预览和截图验收。Agent 不是只交一段 JSON，而是能看到结果、定位帧、调整参数。

这对任何 skill 都是提醒：只写生成步骤不够，最好写清验证方式。

### 3. Skill 可以承载长期状态

Teach skill 把目录里的 `MISSION`、课程、学习记录和参考资料变成状态。这说明 skill 不一定只做一次输出，也可以定义“怎样持续维护一个工作空间”。

但越是有状态，越要写清边界：哪些文件能改，哪些记录能公开，哪些内容不能进入仓库。

### 4. Skill 市场会越来越杂，审计比收集更重要

AIHot 上出现的很多 skill 都有趣，但不是每个都应该立刻安装或推荐。新手最容易犯的错误是把 skill 当插件市场乱装。

公开推荐前至少要确认：

1. 来源是谁。
2. 是否有公开仓库或文档。
3. `SKILL.md` 是否要求读取敏感数据。
4. 是否包含可疑 URL、下载脚本或上传行为。
5. 是否和你现有 workflow 重复。

## 公开分享时怎么写

把一个 skill 放进公开清单时，至少写 5 列：

| 字段 | 写什么 |
|---|---|
| Skill 名称 | 名称或公开项目链接 |
| 作者 / 来源 | 能确认就标注，不能确认就写本地整理 |
| 解决什么问题 | 一句话任务，不写空泛能力 |
| 为什么值得分享 | 它解决了什么真实痛点 |
| 公开边界 | 隐私、版权、账号、权限、验证提醒 |

本仓库的 [Skills 专题导览](../skills/README.md) 就按这个标准维护。

## 最小可用模板

```markdown
# [Skill 名称]

## 适用场景

- 当用户要求 [任务 A] 时使用。
- 当用户提供 [输入类型]，并希望得到 [输出类型] 时使用。
- 当任务需要 [特定工具 / 流程 / 验收方式] 时使用。

## 不适用场景

- 不用于 [相近但不同的任务]。
- 不用于 [高风险或需人工确认的动作]。
- 不用于 [隐私、版权、账号或未授权材料]。

## 工作流

1. 确认输入和边界。
2. 执行核心步骤。
3. 生成产物。
4. 做验收。
5. 输出下一步建议。

## 验收

- 产物存在。
- 来源可追溯。
- 风险动作已确认。
- 没有泄露隐私。
```
