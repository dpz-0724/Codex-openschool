# Agent 新工具 / Skill 安装前审计清单：别被热榜带着装

## 审计信息

- 整理时间：2026-06-12。
- 来源：AIHot 最近 7 天精选条目作为发现入口，补充 Replit、GitHub、Claude、xAI、Alibaba 等公开一手来源。
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 适合读者：经常在 AIHot、X、GitHub、公众号或朋友圈看到 Agent 工具 / skill 推荐，但不知道该不该安装的新手。
- 目标：给出一张安装前可复制的审计表，帮助读者判断“能不能用、值不值得用、怎么安全试用”。
- 隐私处理：不包含本地路径、账号、客户材料、浏览器登录态、密钥、私有 skill 源码、未公开项目或聊天记录。

## 先说结论

看到一个新的 Agent 工具、CLI、插件、skill 或提示词，不要先问“它火不火”，先问这 6 个问题：

```text
它解决什么重复问题？
来源是否可靠？
安装会拿到什么权限？
是否需要密钥、账号或浏览器登录态？
产物能不能验证？
出了问题能不能停下或回滚？
```

如果这 6 个问题答不出来，先不要装进日常工作流。可以收藏观察，但不要给它真实项目、真实账号或私有数据。

## 1. 为什么新手容易被新工具带偏

AI Agent 工具更新太快，热榜里常见几类信息：

| 类型 | 看起来很诱人 | 真正要问 |
|---|---|---|
| 新 CLI | “一键部署”“自动修复”“终端 Agent” | 会改哪些文件？会访问哪些账号？失败怎么回滚？ |
| 新 skill | “安装后 Agent 更聪明” | 触发场景是否清楚？和现有 skill 是否重复？ |
| 新插件 / marketplace | “一键接入生态” | 插件来源、权限、版本和更新机制是否可控？ |
| 新提示词 | “复制即用” | 是否只适合作者场景？有没有禁止动作和验收标准？ |
| 新自动化能力 | “自动浏览、自动发布、自动审查” | 外部动作是否需要人工确认？有没有日志和证据？ |

新手最常见的问题不是“工具不够多”，而是还没建立审计习惯，就把工具接进了真实工作流。

## 2. 三档处理：收录、观察、拒绝

看到新工具时，先不要急着写推荐。按三档处理：

| 档位 | 条件 | 处理 |
|---|---|---|
| 收录 | 有稳定公开入口，解决具体问题，输入输出清楚，能写边界 | 可以进入工具清单或常青教程 |
| 观察 | 只有社交摘要、安装门槛不明、权限不明、还没看到可靠文档 | 先放观察池，不推荐新手安装 |
| 拒绝 | 需要私有账号、绕过限制、读取敏感数据、来源不明、没有回滚方式 | 不放公开主推 |

这也是本仓库维护 `skills/README.md` 的基本标准：宁可少推荐，也不要把普通项目、重复能力或未经验证的热闹条目塞进主清单。

## 3. 安装前 12 项审计表

把下面这张表复制到你的知识库或 issue 里，每个新工具都过一遍。

| # | 审计项 | 要回答的问题 | 不通过信号 |
|---:|---|---|---|
| 1 | 真实问题 | 它解决哪个重复任务？ | 只能说“很酷”“很强”，说不出具体任务 |
| 2 | 来源 | 有官网吗、GitHub、官方文档或原作者说明吗？ | 只有二手转述或截图 |
| 3 | 作者 / 维护者 | 作者是谁，最近是否维护？ | 仓库长期不更新、issue 大量无人处理 |
| 4 | 安装方式 | 安装命令是否清楚、可卸载、可锁版本？ | 只给一键脚本，不说明会写哪里 |
| 5 | 权限 | 会读写哪些文件、调用哪些系统能力？ | 默认要求全盘、浏览器登录态、后台账号 |
| 6 | 密钥 | 是否需要 LLM token、平台 API key、cookie？ | 要求把 token 写进公开文件或聊天 |
| 7 | 输入 | 它需要什么材料才能工作？ | 必须上传客户材料、私有代码或未脱敏聊天 |
| 8 | 输出 | 产物是什么，能否人工审查？ | 只说“已完成”，没有文件、diff、日志、链接 |
| 9 | 验收 | 有没有本地预览、测试、lint、截图或回放？ | 无法判断结果是否正确 |
| 10 | 重复度 | 和已有工具是否重复？ | 只是换名字，能力和现有工具高度重叠 |
| 11 | 失败处理 | 失败后能否停止、回滚、保留证据？ | 会继续重试高风险动作 |
| 12 | 公开边界 | 能否脱敏后公开分享？ | 一讲清价值就必须暴露私有路径、账号或业务数据 |

这张表的目的不是让你变慢，而是避免把工具债带进工作流。

## 4. 用近期例子理解审计点

下面不是给所有工具背书，而是说明“看到不同类型动态时应该审什么”。

| 动态类型 | 例子 | 先看什么 | 新手可学 |
|---|---|---|---|
| Skill 机制 | Replit Agent Skills | description、触发边界、不适用场景 | skill 不是宣传语，是按任务加载的操作手册 |
| 终端智能 | GitHub Copilot CLI + Language Server | 是否能利用项目语义、是否仍需人工审查 | 终端 Agent 不能只会跑命令，还要理解代码结构 |
| 长任务 Agent | Claude Managed Agents | 定时运行、环境变量、运行日志、权限边界 | 长任务关键是环境和审计，不是提示词更长 |
| 插件生态 | xAI Grok Build Plugin Marketplace | 插件来源、版本、权限、pinning | marketplace 方便，但供应链和权限更重要 |
| 专用审查工具 | alibaba/open-code-review | diff 范围、行级评论、LLM 配置和密钥 | 专用工具适合重复审查，不替代负责人判断 |
| 视觉/内容 skill | orange-line-illustration、Text-To-Lottie | 版权、输出格式、本地预览 | 创作类 skill 也要有授权和验收 |

如果一个条目只有社交平台热度，没有稳定公开入口、安装说明和安全边界，就先别主推。

## 5. 四类工具的重点审计

### 5.1 CLI / 本地命令行工具

重点看：

- 会不会修改当前仓库之外的文件。
- 是否需要管理员权限。
- 是否会自动提交、推送、部署或发布。
- 有没有 dry run、preview、diff、rollback。
- 能否用测试命令验证结果。

试用方式：

```text
先在临时仓库或复制项目里跑。
先只给公开样例数据。
先禁用自动发布、自动 push、自动删除。
每次执行前后看 git status 和 diff。
```

### 5.2 Skill / Prompt

重点看：

- description 是否写清触发场景。
- 是否写了“不要使用此 skill 的场景”。
- 输入输出是否稳定。
- 是否和已有 skill 重复。
- 是否包含危险默认动作。

好 skill 应该像任务说明书，而不是能力海报。

### 5.3 浏览器 / 桌面自动化

重点看：

- 是否使用你的日常浏览器登录态。
- 是否会访问账号后台、支付、发布、删除页面。
- 是否会批量抓取或触发平台风控。
- 是否保留操作日志和截图证据。
- 是否能在关键动作前停下来确认。

默认规则：

```text
读取公开页面：低风险。
操作登录态页面：中高风险。
发布、付款、删除、改账号：必须人工确认。
读取 cookie、密钥、浏览器 profile：默认禁止。
```

### 5.4 插件 / Marketplace

重点看：

- 插件是否有官方入口或可信维护者。
- 是否能锁定版本。
- 是否有权限说明。
- 更新是否会自动改变能力边界。
- 是否能禁用、卸载或回退。

插件生态越方便，越要关注供应链和权限。

## 6. 10 分钟试用流程

第一次试一个新工具，不要直接上真实项目。按这个流程做：

```text
1. 找到一手来源：官网 / GitHub / 官方文档。
2. 读 README 的安装、权限、配置和卸载部分。
3. 建一个临时目录或样例仓库。
4. 准备公开、低风险输入。
5. 先跑只读命令或 dry run。
6. 看输出是否有文件、diff、日志、截图或可验证链接。
7. 故意给一个错误输入，观察它如何失败。
8. 写下：适合什么、不适合什么、需要人工确认什么。
9. 决定：收录 / 观察 / 拒绝。
10. 如果收录，再写进自己的工具清单或 skill 说明。
```

这 10 分钟能挡住大部分“看起来很强但不适合你”的工具。

## 7. 给 Agent 的审计提示词

以后看到一个新工具链接，可以直接让 Agent 先做审计，而不是让它马上安装：

```text
请审计这个 Agent 工具 / skill 是否适合进入我的日常工作流。

输入：
- 工具链接：
- 我想解决的问题：
- 我的限制：不上传私有代码、不读取密钥、不自动发布、不碰真实账号后台。

请只基于一手来源和仓库内容判断，不要根据宣传语下结论。

输出：
1. 它解决的真实问题。
2. 官方入口、作者和维护状态。
3. 安装方式和写入位置。
4. 需要的权限、账号、密钥或环境变量。
5. 输入、输出和验收方式。
6. 和我已有工具是否重复。
7. 风险分级：低 / 中 / 高。
8. 试用建议：收录 / 观察 / 拒绝。
9. 如果收录，写一句适合放进 skills 目录的说明。

禁止：
- 不要直接安装。
- 不要运行一键脚本。
- 不要读取 `.env`、cookie、浏览器 profile、私有仓库或客户材料。
- 不要把二手摘要当成最终事实。
```

## 8. 什么时候可以公开推荐

一个工具进入公开分享前，至少要满足：

1. 有稳定公开入口。
2. 能解决一个具体问题。
3. 输入、输出、验收方式清楚。
4. 风险边界能讲明白。
5. 不需要暴露私有材料才能证明价值。
6. 来源、作者或维护者可说明。
7. 和已有推荐不重复。

如果只能写出“最近很火”“据说很好用”“很多人转发”，就不要放到公开主清单。

## 10. 一句话结论

AI 时代不是谁装的工具多谁厉害，而是谁能把工具放进可验证、可停止、可回滚的工作流里。新工具可以试，但先审计，再安装；先低风险样例，再真实任务；先看证据，再写推荐。
