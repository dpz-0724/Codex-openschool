# Agent 工具调用选型：什么时候用 MCP，什么时候用 CLI

## 审计信息

- 来源：本地 wiki 中关于 MCP vs CLI 工具调用的决策页和来源摘要；公开版已脱敏、重写，并补充官方 MCP 文档与 Anthropic 工程文章。
- 作者：本地整理；公开版由 Codex 改写。
- 适合读者：已经知道 Agent 可以调用工具，但不确定该接 MCP、CLI、API 还是浏览器自动化的新手。
- 目标：帮助读者在做第一个 Agent 工具层时少走弯路，先选对接口形态，再谈复杂架构。
- 隐私处理：不包含本地路径、账号、后台数据、密钥、真实业务脚本或私有 MCP Server 配置。

## 1. 先把问题说清楚

新手很容易把“让 Agent 会用工具”理解成“必须上 MCP”。这不准确。

MCP、CLI、API、浏览器自动化都可以让 Agent 做事。区别不在于谁更高级，而在于它们解决的问题不同：

| 接口形态 | 新手理解 | 更适合什么 |
|---|---|---|
| MCP | 给 Agent 接一套标准化工具协议 | 长期运行、多工具共享、需要权限和审计的工具层 |
| CLI | 让 Agent 调一条命令或脚本 | 快速验证、本地文件处理、开发者工具、一次性任务 |
| API | 让 Agent 请求一个明确服务接口 | 服务端集成、业务系统、稳定数据读写 |
| 浏览器自动化 | 让 Agent 像人一样操作网页 | 没有 API、需要看页面、需要登录态或人工确认的网页任务 |

这篇只重点讲 MCP 和 CLI，因为它们是 Agent 工具层里最容易被混淆的一组。

## 2. MCP 解决什么

MCP 全称是 Model Context Protocol。官方文档把它定义为连接 AI 应用和外部系统的开放标准。它让 AI 应用可以连接数据源、工具和 workflow。

对新手来说，可以这样理解：

- MCP Server：把某个外部能力包装成标准工具。
- MCP Client：Agent 或 AI 应用通过标准协议发现并调用这些工具。
- Tool schema：告诉模型有哪些工具、参数怎么填、返回什么。
- 权限和审计：让工具调用更可控、更容易记录。

MCP 的强项是标准化。多个 Agent、多个应用、多个工具都要长期共用时，标准化就很有价值。

## 3. CLI 解决什么

CLI 是命令行工具。它没有 MCP 那么标准，但它有三个现实优势：

1. 快。很多工具本来就有命令行入口。
2. 省。通常不需要把一大堆工具 schema 放进上下文。
3. 熟。模型对 `git`、`node`、`python`、`ffmpeg`、`pandoc`、`rg`、`curl` 这类命令模式有大量训练经验。

如果任务只是“把一个 Markdown 转成 PDF”“查一下 git log”“批量重命名几个文件”“跑一个本地检查脚本”，直接 CLI 往往比临时写 MCP Server 更务实。

## 4. 快速判断表

| 你的情况 | 优先选择 | 原因 |
|---|---|---|
| 工具会修改文件、访问敏感数据、触发外部动作 | MCP 或受控 API | 需要权限边界、输入校验、审计和人工确认 |
| 工具要被多个 Agent 长期复用 | MCP | 标准化接口降低协作成本 |
| 工具需要细粒度权限 | MCP | 可以按工具、资源、用户做访问控制 |
| 只是本地一次性处理文件 | CLI | 成本低、验证快 |
| 已经有成熟命令行工具 | CLI | 不要为了“Agent 化”重复封装 |
| 上下文窗口很紧张 | CLI 或按需加载工具 | MCP schema 可能占用大量上下文 |
| 要接企业用户数据或管理动作 | MCP + Authorization / 受控 API | 需要身份、授权、日志和治理 |
| 还在验证想法 | CLI / 脚本 | 先确认任务真的有价值，再工程化 |

## 5. 决策流程

```text
这个工具会不会读写敏感数据、修改系统、发起外部动作？
├─ 会
│  ├─ 是否需要长期复用、权限控制、审计？
│  │  ├─ 是：优先 MCP 或受控 API
│  │  └─ 否：先做受限脚本 + 人工确认，不要直接放权
│  └─ 每次执行前都要展示输入、影响范围和回滚方式
└─ 不会
   ├─ 是否已有成熟 CLI？
   │  ├─ 是：优先 CLI
   │  └─ 否：继续判断
   ├─ 是否多个 Agent 长期共享？
   │  ├─ 是：考虑 MCP
   │  └─ 否：先用脚本或 API
   └─ 是否只是网页上才能完成？
      ├─ 是：考虑浏览器自动化
      └─ 否：保持最简单可验证方案
```

## 6. 新手最容易踩的坑

### 坑一：把 MCP 当作万能接口

MCP 是协议，不是质量保证。一个 MCP Server 如果没有做好输入校验、权限控制、日志和输出净化，仍然可能很危险。

官方 MCP tools 规范里也强调：Server 需要校验输入、做访问控制、限流、清理输出；Client 对敏感操作应提示用户确认、展示工具输入、验证结果、设置超时并记录调用。

### 坑二：为了标准化牺牲简单性

如果一个任务用一条命令就能稳定完成，不必一开始就做 MCP Server。先用 CLI 验证真实需求，再决定是否封装。

### 坑三：把所有工具 schema 一次性塞给模型

工具越多，模型不一定越强。Anthropic 在 code execution with MCP 的文章里提到一个方向：让 Agent 在代码执行环境里按需加载和调用工具，减少不必要的上下文占用。

新手可以先记住一句话：

> 工具不是越多越好，当前任务需要什么才给什么。

### 坑四：忽略权限和确认

只要工具能改文件、发请求、上传内容、删除数据、付款、发布，就不能让 Agent 静默执行。至少要做到：

- 执行前说明要做什么。
- 展示关键参数。
- 标出影响范围。
- 高风险动作等用户确认。
- 执行后保留日志或结果证据。

## 7. 四个实战例子

### 例子一：整理本地 Markdown

任务：把一批 Markdown 做链接检查、标题规范化、目录更新。

建议：

- 优先 CLI / 脚本。
- 用 `rg` 查链接，用脚本检查路径，用 Markdown linter 做格式检查。
- 只有当这套能力要被多个 Agent 反复调用时，再考虑封装成 MCP。

原因：低风险、本地文件、命令链成熟，CLI 成本更低。

### 例子二：连接公司知识库

任务：让 Agent 查询内部文档、按权限返回摘要。

建议：

- 优先 MCP 或受控 API。
- 要有身份认证、权限过滤、审计日志和结果脱敏。
- 不要让 Agent 直接拿数据库账号随意查。

原因：涉及用户数据、组织权限和审计要求。

### 例子三：生成 PPT

任务：从公开资料生成一版课程 PPT。

建议：

- 资料抓取可以用浏览器自动化或网页转 Markdown。
- 中间整理可以用 CLI / 脚本。
- PPT 生成可以用已有 skill 或工具链。
- 如果团队长期共用，可把“资料读取、结构化、导出、验收”拆成标准工具。

原因：这是混合工作流，不必强行只选一种接口。

### 例子四：自动发布内容

任务：让 Agent 把文章发布到平台。

建议：

- 默认不要全自动发布。
- 草稿生成、排版、封面建议可以自动化。
- 登录、发布、删除、修改公开内容必须人工确认。
- 如果平台提供 API，优先用受控 API；没有 API 时再考虑浏览器自动化。

原因：发布是外部可见动作，错误成本高。

## 8. 给新手的最小实践

用 30 分钟做一个小练习：

1. 选一个低风险任务，例如“检查一个 Markdown 文件里是否有本地绝对路径”。
2. 先用 CLI 完成，例如用 `rg` 搜索可疑模式。
3. 让 Agent 解释命令做了什么、输出说明了什么。
4. 再问：如果这件事要给团队长期使用，是否值得封装成 MCP？
5. 写下判断理由：风险、复用频率、权限、审计、上下文成本。

练习产物：

- 一条可复用命令。
- 一段工具说明。
- 一张 MCP vs CLI 选择表。
- 一条安全边界说明。

## 9. 工具说明卡模板

以后你给 Agent 增加任何工具前，先写这张卡：

```markdown
## 工具名称

## 解决的问题

## 输入

## 输出

## 适合场景

## 不适合场景

## 风险

## 是否需要人工确认

## 为什么选择 MCP / CLI / API / 浏览器自动化
```

如果这张卡写不清，先不要把工具接给 Agent。

## 10. 一句话结论

MCP 适合长期、标准化、可审计、需要权限边界的工具层；CLI 适合轻量、快速、低风险、已有命令行能力的任务。新手不要追接口名词，先追任务闭环、风险边界和可验证结果。
