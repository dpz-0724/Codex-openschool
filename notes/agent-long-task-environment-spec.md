# Agent 长任务环境规格模板：先圈环境，再让 Agent 自主跑

## 审计信息

- 整理时间：2026-06-12。
- 来源：AIHot 最近 7 天精选里的 EurekAgent、OpenAI 收购 Ona 的官方公告、EurekAgent arXiv 摘要，以及本仓库 Enterprise Agent Harness 笔记。
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 适合读者：已经会让 Agent 改文件、查资料、跑命令，但想把任务交给它跑 30 分钟以上的人。
- 目标：给新手一份可复制的长任务环境规格，先写清权限、产物、预算、人工介入和回放方式，再开始执行。
- 隐私处理：不包含本地路径、账号、客户材料、真实聊天记录、私有仓库、密钥或未发布项目。

## 先说结论

长任务 Agent 最容易失控的原因，不是模型不够聪明，而是环境没有设计好。

你不能只说：

```text
帮我持续维护这个仓库，自己看着办。
```

更稳的说法是：

```text
你可以读取哪些资料，可以修改哪些文件，哪些动作必须停下来问我，每一步留下什么证据，最多花多少时间，什么时候停止，失败后怎么回放。
```

这就是环境规格。

## 为什么要先写环境规格

AIHot 里的 EurekAgent 把“环境工程”放到了核心位置：Agent 能不能可靠探索，不只取决于 prompt，还取决于资源、约束和接口。论文摘要把它拆成四类：权限工程、产物工程、预算工程、人在回路工程。

OpenAI 在 2026-06-11 公告中也提到，长时间运行的 Codex 需要安全、持久、客户可控的云端环境。公告里强调的不只是“能跑更久”，还包括访问边界、凭据范围、活动日志和审查流程。

对新手来说，可以把这两件事压缩成一句话：

> 长任务开始前，先把 Agent 的活动空间设计出来。

## 什么时候需要这张表

下面这些任务，建议先写环境规格：

| 场景 | 为什么需要 |
|---|---|
| 任务预计超过 30 分钟 | 中途状态容易丢，必须有 checkpoint |
| 会改多个文件 | 需要限制可写范围和回滚方式 |
| 会联网查资料 | 需要来源、版权、登录和反爬边界 |
| 会运行脚本或测试 | 需要命令白名单和预算上限 |
| 会发布、提交、部署或发消息 | 必须有人类确认点 |
| 会处理未脱敏资料 | 默认不应该进入公开环境 |
| 会连续多轮自主决策 | 需要停止条件和复盘日志 |

如果只是改错别字、回答概念问题、整理一小段公开资料，不需要完整环境规格。用 [10 行 Spec](agent-coding-10-line-spec-template.md) 就够了。

## 每一块怎么写

### 1. 任务边界

任务边界回答三个问题：

1. Agent 可以自己推进到哪里。
2. 哪些问题必须先问人。
3. 发现任务变大时怎么办。

示例：

```text
允许自主推进的范围：整理公开资料，写 Markdown 草稿，更新目录链接。
需要先请示的问题：新增目录结构、删除旧内容、引入未验证外部项目。
发现范围变大时怎么处理：先写出新增范围和原因，不直接执行。
```

### 2. 权限边界

权限边界要写得具体，不要只写“注意安全”。

| 权限项 | 好写法 | 弱写法 |
|---|---|---|
| 只读范围 | `notes/`、公开 URL、官方文档 | 可以看相关资料 |
| 可写范围 | 只能新增 `notes/*.md` 和更新目录 | 可以改仓库 |
| 禁止动作 | 不读 `.env`、不处理聊天记录、不发布平台内容 | 不要乱来 |
| 必须确认 | 提交代码、删除文件、改远端配置 | 重要操作问我 |

如果你不能清楚写出只读范围和可写范围，就不要让 Agent 长时间自主跑。

### 3. 产物契约

产物契约解决一个常见问题：Agent 跑了很久，最后只说“我做完了”，但没有证据。

推荐要求 Agent 留下这些产物：

| 产物 | 用途 |
|---|---|
| 计划 | 开始前确认方向 |
| 变更清单 | 知道改了什么 |
| 来源列表 | 追溯事实 |
| 决策记录 | 解释为什么这样取舍 |
| 检查结果 | 证明不是只靠感觉 |
| 最终交付物 | 读者真正要用的东西 |

最小要求：

```text
每个外部事实必须有来源链接。
每次文件改动必须能在 diff 里看到。
每个“完成”必须有检查项或证据。
```

### 4. 预算和停止条件

预算不是只管钱，也包括时间、工具调用、搜索范围和注意力。

示例：

```text
最长运行时间：本轮最多 60 分钟。
最多联网来源数量：最多核 5 个一手来源。
最多命令次数：不超过 20 条本地命令。
停止条件：找不到一手来源、发现隐私风险、需要登录、任务范围扩大超过原计划。
预算快耗尽时怎么汇报：列出已完成、未完成、阻塞点和下一步建议。
```

停止条件要提前写。如果等 Agent 已经跑偏后再叫停，通常已经产生了无用输出。

### 5. 人在回路

人在回路不是每一步都问人。它是把真正高风险、不可逆、会影响方向的点拎出来。

必须停下来的情况：

- 要发布、提交、部署、发消息。
- 要读取账号后台、聊天记录、客户材料或密钥。
- 要删除、覆盖、移动大量文件。
- 要扩大任务范围。
- 找不到可靠来源却想继续推断。
- 预算快耗尽但还没有有效产物。

允许自行决定的情况：

- 改正文措辞。
- 补公开来源链接。
- 修 Markdown 内部链接。
- 增加无风险的检查表。
- 重排笔记结构但不改变主题。

这样既不会让 Agent 每一步都卡住，也不会把高风险动作放飞。

### 6. 回放和回滚

长任务必须能回放。回放不是为了追责，而是为了下一轮能变好。

最小回放材料：

```text
输入：原始需求、确认过的环境规格、使用的公开来源。
过程：计划、关键决策、命令输出、检查结果。
输出：最终文件、diff、链接、截图或报告。
失败点：哪一步没有证据、哪里超范围、哪里需要人确认。
```

如果任务涉及文件改动，还要写：

```text
回滚方式：只用 git diff / git status 判断改动；不得擅自 reset 或 checkout 用户改动。
```

## 示例：维护一个公开知识库仓库

这是适合本仓库的一个脱敏示例。

```text
# Agent 长任务环境规格

任务名称：维护 Codex OpenSchool 的一篇新手教程。
任务类型：公开知识库内容更新。
目标读者 / 使用者：AI Agent 新手。
最终交付物：一篇新的 Markdown 笔记，更新 README 和 notes/README.md。
成功标准：读者打开首页能找到新笔记；新笔记有明确受众、模板、边界、来源和自检；无隐私泄露。
明确不做：不上传私有 skill 源码，不处理聊天记录，不自动发布社交平台，不追求凑数量。

## 1. 任务边界
允许自主推进的范围：从 AIHot 和一手来源中筛选一个主题，写成公开教程。
需要先请示的问题：删除旧内容、改仓库定位、发布到外部平台。
发现范围变大时怎么处理：先更新计划，说明为什么需要扩大。

## 2. 权限边界
只读范围：仓库 Markdown 文件、公开网页、官方文档、论文摘要。
可写范围：新增 notes/*.md，更新 README、notes/README.md、digests 或 GROWTH_PLAYBOOK。
允许使用的工具：搜索、公开网页读取、git diff、Markdown 链接检查。
允许运行的命令：git status、git diff、rg、链接检查脚本。
禁止动作：读取密钥、聊天记录、账号后台、客户材料、本地私有路径。
必须人工确认的动作：删除文件、改远端配置、发布到社交平台。
隔离方式：只在当前仓库工作，不把本地私有资料复制进公开仓库。

## 3. 产物契约
开始前必须写出的计划：主题、来源、更新文件、审计方式。
每一步必须留下的日志：关键来源、为什么收录、为什么不收录。
每个关键判断必须留下的证据：一手链接或明确标注为本地整理。
中途 checkpoint：完成正文后先查链接和隐私。
最终交付物：Markdown 正文、目录入口、增长手册记录。
不能只用口头总结替代的产物：不能只说“已整理”，必须有文件和链接。

## 4. 预算和停止条件
最长运行时间：本轮 60-90 分钟。
最多联网来源数量：5 个核心来源以内。
最多工具调用 / 命令次数：足够完成链接、隐私和远端验证即可。
成本或 token 上限：不做大规模抓取。
停止条件：来源不可核、涉及隐私、主题和已有内容重复、无法写成正文。
预算快耗尽时怎么汇报：列出已完成文件、未完成风险和下一步。

## 5. 人在回路
必须停下来问人的情况：需要发布外部平台、上传本地私有材料、删除旧内容。
允许自行决定的情况：补目录链接、修标题一致性、改写为新手友好表达。
中途汇报节奏：完成选题和完成正文后各汇报一次。
用户需要审核什么：主题是否值得公开、是否涉及隐私、是否符合频道定位。
用户如何批准 / 驳回 / 修改方向：直接说明保留、删除、改主题或推送。

## 6. 回放和回滚
失败后如何复现：保留来源链接、diff、检查命令结果。
需要保留哪些输入和输出：AIHot 条目、官方来源、最终 Markdown。
如何回滚文件改动：只用 git diff 定位改动；不擅自 reset 用户改动。
如何判断下一轮是否变好：打开首页能找到，读者能复制模板执行。

## 7. 隐私和版权
是否涉及账号、聊天、客户、后台、密钥或本地路径：不涉及。
是否使用公开资料：是，只使用公开链接。
是否可能复制长篇原文：不复制，只做转述和模板。
需要脱敏的内容：本地路径、账号、私有流程、未发布项目。
不能进入公开仓库的内容：聊天记录、客户材料、密钥、后台截图。
```

## 可复制提示词：执行前先写环境规格

```text
先不要执行任务。
请根据下面的目标，写一份 Agent 长任务环境规格。

目标：
[粘贴你的任务]

请必须覆盖：
1. 任务边界：允许自主推进什么，哪些要先问我。
2. 权限边界：只读范围、可写范围、允许工具、禁止动作、必须人工确认的动作。
3. 产物契约：计划、日志、证据、checkpoint、最终交付物。
4. 预算和停止条件：最长时间、最多来源/命令/工具调用、何时停止。
5. 人在回路：什么情况必须停下来问我，什么情况可以自己决定。
6. 回放和回滚：失败后如何复现，文件改动如何回滚。
7. 隐私和版权：是否涉及账号、聊天、客户材料、密钥、本地路径或长篇原文。

输出后请给出判断：
- 可以开始执行
- 需要我确认后再执行
- 不建议执行，原因是...
```

## 可复制提示词：执行中汇报 checkpoint

```text
请按环境规格做一次 checkpoint 汇报。

输出：
1. 当前已完成什么。
2. 已产生哪些文件、链接、命令输出或证据。
3. 是否仍在只读范围和可写范围内。
4. 是否触发必须人工确认的动作。
5. 已消耗多少时间 / 来源 / 命令 / 工具调用。
6. 是否接近预算或停止条件。
7. 下一步准备做什么。
8. 我需要批准、驳回或修改什么。

如果没有证据，不要说“已完成”。
```

## 可复制提示词：执行后做环境审计

```text
请审计这次长任务是否遵守环境规格。

输出表格：
| 环境规格项 | 证据 | 是否符合 | 需要修复 |

必须检查：
1. 是否超出任务边界。
2. 是否读取或写入了未授权范围。
3. 是否执行了禁止动作。
4. 是否把必须人工确认的动作自动执行了。
5. 是否留下了计划、日志、证据和最终交付物。
6. 是否超出预算或忽略停止条件。
7. 是否存在隐私、版权、密钥、本地路径泄露。
8. 失败后是否能回放。

最后给出结论：
- 通过，可以归档。
- 有小问题，需要补证据。
- 不通过，需要回滚或人工介入。
```

## 常见失败模式

### 1. 只写目标，不写权限

```text
目标：帮我自动整理所有资料。
```

这会让 Agent 不知道能读什么、不能读什么。

改成：

```text
只读范围：公开网页和已脱敏 Markdown。
禁止动作：不读取聊天记录、客户资料、账号后台、密钥和本地私人目录。
```

### 2. 只有最终交付，没有中途产物

长任务没有中途产物，失败时就无法判断哪里出了问题。

至少要求：

```text
开始前写计划。
每个关键判断留来源。
每 30 分钟做 checkpoint。
最终给出文件列表和验证结果。
```

### 3. 停止条件太晚

错误写法：

```text
做不下去再说。
```

改成：

```text
如果连续 2 次找不到一手来源、需要登录、发现隐私材料、或任务范围扩大，就停止并汇报。
```

### 4. 人在回路变成“每步确认”

每步确认会让 Agent 没法工作。正确做法是只拦高风险动作：

```text
低风险：改写文字、补公开来源、修 Markdown 链接，可以自主做。
高风险：删除文件、提交远端、发布平台、读未脱敏资料，必须确认。
```
