# 个人知识库搭建：让 Agent 维护一个会生长的 Wiki

## 审计信息

- 来源：本地 notebook 中关于知识库重构、知识编译工作流和 Notebook Knowledge OS 的多篇笔记。
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 脱敏处理：已删除本地路径、实例名、端口、私人项目、聊天记录、账号后台、客户材料和不可公开数据。
- 适合读者：想用 Obsidian、Markdown、Codex 或其他 Agent 工具搭建长期个人知识库的人。
- 审计结论：通过。它不是标题清单，而是一套可执行的知识库维护方法。

## 1. 核心判断

个人知识库最常见的问题，不是资料不够多，而是资料没有变成可复用的知识。

很多人的知识库最后会变成三个东西：

- 文件夹越来越多。
- 标签越来越细。
- 收藏越来越长。

但真正需要使用时，仍然会遇到“找不到、想不起、找到了也用不上”的问题。

更好的心智模型是：不要把知识库当成资料仓，而要把它当成一个由 Agent 协助维护的长期 wiki。

这个 wiki 的目标不是保存一切，而是把原始材料持续编译成：

- 能回答问题的页面。
- 能复用的方法。
- 能回放的决策。
- 能长期成立的原则。
- 能追溯来源的证据链。

## 2. 推荐架构

一个可长期运行的个人知识库，可以分成五层：

```text
raw/       原始资料层：文章、想法、AI 输出、摘录、临时笔记
compile/   中间编译层：每日小结、待整理材料、临时综合稿
wiki/      正式知识层：问题、方法、决策、原则、来源摘要
outputs/   输出沉淀层：高价值问答、报告、文章、课程稿
archive/   归档层：过时材料、重复材料、低复用价值材料
```

关键原则：

- `raw/` 是来源，不直接当结论。
- `wiki/` 是正式知识，不存未经判断的原始材料。
- `outputs/` 不是一次性成稿，里面有价值的结论要回写到 `wiki/`。
- `archive/` 不是垃圾桶，而是让主知识库保持清爽的治理层。

如果只想最小启动，可以先保留四个东西：

```text
raw/
wiki/
wiki/index.md
wiki/log.md
```

`index.md` 负责让人和 Agent 快速找到知识，`log.md` 负责记录知识库如何演化。

## 3. 正式知识页应该回答什么

正式 wiki 不建议按“文件夹好不好看”来组织，而应按知识的用途来组织。

推荐把正式知识页分成六类：

| 类型 | 它回答的问题 | 例子 |
|---|---|---|
| Question | 我在解决什么问题？ | 如何让 AI 生成的笔记不污染知识库？ |
| Method | 这类问题通常怎么做？ | 新资料进入知识库的 ingest 流程 |
| Decision | 为什么选这个方案？ | 为什么知识库主轴用问题、方法、决策，而不是普通目录分类 |
| Principle | 为什么这个判断长期成立？ | 知识库要优化复用和判断，而不是优化收藏量 |
| Source | 这批原始材料说了什么？ | 某篇文章、某次复盘、某组 AI 输出的来源摘要 |
| Output | 这次查询产出了什么可复用结果？ | 一份综合回答、一篇文章提纲、一份课程脚本 |

这套分类的价值在于：它逼迫每一页都承担明确职责。

一页如果既不是问题、也不是方法、也不是决策、也不能支撑原则，通常不应该进入正式 wiki。

## 4. Ingest：新资料如何入库

不要把“整理一下”当作入库流程。它太含糊，容易让 Agent 生成一堆漂亮但没用的页面。

更稳定的 ingest 流程是：

1. 新资料先进入 `raw/`。
2. 为资料写来源摘要。
3. 做入库门禁判断。
4. 只更新真正相关的正式页。
5. 更新 `wiki/index.md`。
6. 追加 `wiki/log.md`。

来源摘要建议回答四个问题：

```text
1. 这是什么内容？
2. 核心观点是什么？
3. 它在回答什么问题？
4. 它未来可以用到哪里？
```

然后做四问门禁：

| 门禁问题 | 判断标准 |
|---|---|
| 可复用吗？ | 以后是否还会遇到类似场景 |
| 能抽象成方法或原则吗？ | 是否能从个案提炼出通用逻辑 |
| 能连接到已有知识吗？ | 是否能补强、修正或挑战现有页面 |
| 三个月后还有价值吗？ | 是否不是短期噪音或一次性情绪 |

建议至少两个问题回答“是”，才进入正式 wiki。否则就留在 `raw/` 或进入 `archive/`。

## 5. Query：提问时如何使用知识库

问 Agent 问题时，不要让它直接凭模型记忆回答。要明确要求它先查 wiki。

权限边界也要说清楚：默认只让 Agent 读取正式 wiki 和已脱敏来源摘要，不要让它自动扫私人 raw、聊天导出、账号后台或项目材料。

推荐流程：

1. 先读 `wiki/index.md`。
2. 优先读相关的 `questions/`、`methods/`、`decisions/`。
3. 需要证据时再读 `sources/`。
4. 给出带来源的结构化回答。
5. 如果回答有长期价值，写入 `outputs/`。
6. 如果回答产生了新方法、新决策或新原则，再回写正式 wiki。

核心原则是：好的 query 不应该死在聊天记录里。

如果一个回答未来还会被用到，它就应该成为知识库的新资产。

## 6. Lint：定期给知识库做健康检查

知识库不是整理一次就结束。它需要周期性 lint。

建议检查这些问题：

- 有没有只有标题、没有方法的页面？
- 有没有方法页缺少适用边界？
- 有没有决策页的前提已经过时？
- 有没有来源摘要长期没有连接到正式知识？
- 有没有反复出现但没有正式页面的关键概念？
- 有没有多个页面在表达互相矛盾的判断？
- 有没有高价值 output 没有回写到 wiki？
- 有没有孤儿页、断链、重复页？

这一步很适合交给 Agent 做初筛，人只负责拍板。

## 7. 三个实用提示词

### 入库提示词

```text
帮我 ingest 这份资料。

要求：
1. 先写来源摘要：这是什么、核心观点、回答什么问题、未来可用场景。
2. 做四问门禁：可复用吗、能抽象成方法或原则吗、能连接已有知识吗、三个月后还有价值吗。
3. 如果通过门禁，只更新真正相关的正式 wiki 页面，不要机械拆成很多薄页。
4. 更新 index 和 log。
5. 标出哪些内容不能直接当结论，需要保留在来源层。
```

### 查询提示词

```text
请先查我的 wiki 再回答这个问题。

流程：
1. 先读 index，判断相关的问题、方法、决策、原则页面。
2. 必要时读取来源摘要。
3. 回答时标明依据来自哪些页面。
4. 如果这次回答有长期复用价值，请整理成 output。
5. 如果产生了新方法、新决策或新原则，请建议是否回写正式 wiki。
```

### 健康检查提示词

```text
请对这个知识库做一次 lint。

重点检查：
1. 孤儿页、断链、重复页。
2. question 缺 method。
3. method 缺适用边界。
4. decision 的假设是否过时。
5. source 是否没有接入正式知识。
6. output 是否有值得回写但未回写的内容。
7. 页面之间是否存在冲突或重复表达。

最后给出：必须修、建议修、暂不处理 三类清单。
```

## 8. 不推荐的做法

| 做法 | 问题 |
|---|---|
| 让 Agent “帮我整理一下” | 目标不清，容易产出漂亮但不可复用的摘要 |
| 把 raw 笔记直接搬进 wiki | 原始想法会冒充稳定结论 |
| 只按概念建页面 | 容易变成百科，而不是行动知识库 |
| 不维护 index | 知识进去了，但人和 Agent 都找不到 |
| 不记录 log | 无法回放知识库为什么变成现在这样 |
| 只存 outputs，不回写 | 好答案留在一次性成稿里，无法长期复利 |

## 9. 最小可执行方案

如果今天就想开始，可以按这个版本搭：

```text
knowledge-base/
├── raw/
│   ├── human/
│   ├── ai/
│   └── external/
├── wiki/
│   ├── questions/
│   ├── methods/
│   ├── decisions/
│   ├── principles/
│   ├── sources/
│   ├── outputs/
│   ├── index.md
│   └── log.md
└── archive/
```

第一周只做三件事：

1. 所有新材料先进 `raw/`。
2. 每天只挑一到三份资料做 ingest。
3. 每周跑一次 lint，把低价值页面挡在正式 wiki 外面。

这套方法真正重要的不是目录，而是门禁、来源链和回写机制。

当知识库能持续回答真实问题、沉淀真实方法、记录真实决策时，它才开始从“资料存储”变成“可复用的个人操作系统”。
