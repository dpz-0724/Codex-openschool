# Agent 代码审查工作流：从 Auto-review 到 Bugbot

## 审计信息

- 来源：AIHot 近期 Agent 精选条目作为发现入口；正文以 Cursor 官方博客和本仓库已有 Agent 审计框架为依据。
- 一手来源：
  - [Cursor: Governing agent autonomy with Auto-review](https://cursor.com/blog/agent-autonomy-auto-review)
  - [Cursor: Bugbot is now over 3x faster, 22% cheaper, and finds 10% more bugs](https://cursor.com/blog/bugbot-updates-june-2026)
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 适合读者：已经开始让 Agent 写代码、改文件、提交 PR，但还没有稳定审查流程的新手。
- 目标：把“让 Agent 写代码”升级成“让 Agent 产生可审查、可回放、可合并的变更”。
- 隐私处理：不包含本地路径、账号、私有仓库、客户材料、密钥、内部 PR、真实业务 diff 或未发布项目。

## 一句话结论

Agent 代码审查不要只发生在 PR 末尾。更稳的做法是三段式：

```text
动作前：判断这一步是否越权。
提交前：检查 diff 是否符合任务和安全边界。
PR 后：让自动审查聚焦新增变更，不重复制造噪音。
```

Cursor 的 Auto-review 说明了第一段：Agent 执行动作前，需要根据用户意图、动作后果和上下文判断风险。

Cursor 的 Bugbot 更新说明了第二、三段：代码推上去之前也可以先跑审查；进入 PR 后，审查应该尽量聚焦新的变更，避免重复反馈。

对新手来说，真正要学的不是某个按钮，而是把 Agent 编程拆成可检查的闭环。

## 为什么只让 Agent “写完再说”不够

很多新手会这样用 Agent：

```text
帮我实现这个功能。
```

然后等 Agent 说“完成了”，再看页面或跑测试。

这个流程的问题是：失败出现得太晚。等你看到错误时，Agent 可能已经改了很多文件，你很难判断：

- 它为什么改这些文件？
- 哪些改动是必须的？
- 哪些改动是顺手重构？
- 有没有碰到权限、密钥、配置、生产数据？
- 测试没跑是因为忘了，还是因为环境不支持？

所以，Agent 代码审查要前移。

不是等它写完才审，而是在三个位置都设置门禁。

## 三段式审查框架

| 阶段 | 核心问题 | 产物 | 失败信号 |
|---|---|---|---|
| 动作前 | 这一步该不该做？ | 工具调用计划、风险判断 | 要读密钥、改生产配置、批量删除、绕过权限 |
| 提交前 | 这个 diff 能不能交出去？ | 变更摘要、测试结果、风险清单 | 改动超范围、无测试、无回滚说明 |
| PR 后 | 审查是否聚焦新增问题？ | Review comment、修复 commit、复审记录 | 重复反馈、旧问题反复出现、自动审查和人工审查脱节 |

这三段不是为了让流程变慢，而是为了让 Agent 的工作更像工程交付。

## 第一段：动作前审查

动作前审查解决的是：Agent 有没有资格做下一步。

Cursor Auto-review 的关键启发是：权限不是一个全局开关。风险取决于动作、上下文、用户意图和失败后果。同一个命令，在不同任务里可能完全不同。

例如：

| 动作 | 低风险场景 | 高风险场景 |
|---|---|---|
| 读取文件 | 读公开仓库里的 README | 读 `.env`、cookie、账号配置 |
| 运行脚本 | 跑本地单元测试 | 执行会删除数据的脚本 |
| 修改配置 | 改示例项目设置 | 改生产部署凭据或 CI secret |
| 提交代码 | 提交自己刚改的文档 | 提交包含未脱敏数据的文件 |

新手可以不用实现一个真正的分类器，但要在提示词里加入类似判断。

### 动作前提示词

```text
在执行任何工具调用前，先判断这一步属于哪类风险：

L0：只读当前公开仓库内容。
L1：修改当前任务明确要求的普通文件。
L2：运行测试、格式化、生成文件，可能影响较多文件。
L3：提交、推送、发布、调用外部服务、影响远端状态。
L4：读取密钥、账号、cookie、生产数据、私有聊天、客户材料，默认禁止。

如果是 L0-L1，可以继续，但要保留证据。
如果是 L2，先说明会影响什么，再执行。
如果是 L3，必须等待我确认。
如果是 L4，直接停止并说明原因。
```

这段提示词的作用不是装饰，而是减少 Agent “顺手做危险动作”的概率。

## 第二段：提交前审查

提交前审查解决的是：这个 diff 是否配得上进入远端。

Cursor Bugbot 更新里一个值得新手学习的点是：审查可以在 push 前发生，而不是等 PR 打开之后才发现问题。

如果你不用 Cursor，也可以把同样思路迁移到 Codex、Claude Code、GitHub Copilot 或普通命令行流程里。

### 提交前最小检查

每次准备提交前，至少让 Agent 输出这 6 项：

```text
1. 改了哪些文件？
2. 每个文件为什么要改？
3. 有没有超出原任务范围？
4. 跑了哪些检查？结果是什么？
5. 还剩哪些风险没有验证？
6. 是否包含隐私、密钥、本地路径、客户材料或未授权内容？
```

如果这 6 项答不清楚，不要提交。

### 提交前审查提示词

```text
请对当前 git diff 做提交前审查。

目标：
- 判断变更是否严格服务于本次任务。
- 找出可能的 bug、越权修改、隐私泄露、测试缺口和文档不一致。
- 不要重写代码，先给审查结论。

输出：
1. 是否建议提交：是 / 否 / 需要补充验证。
2. 主要风险，按严重程度排序。
3. 超范围改动。
4. 必须补的测试或验证命令。
5. 隐私和安全扫描结论。
6. 建议 commit message。
```

这段提示词适合在 `git diff` 后使用，也适合让另一个 Agent 或模型做交叉检查。

## 第三段：PR 后审查

PR 后审查解决的是：自动审查和人工审查如何协同。

Cursor Bugbot 更新里另一个有用点是：审查不应该总是反复看整个 PR。一个 PR 被多次 push 后，如果每次都重新扫所有内容，很容易产生重复反馈，让人分不清哪些是新问题。

所以，PR 后审查要建立两条规则：

1. 新 push 后优先看新增 diff。
2. 已确认的旧问题不要重复制造噪音，除非新改动重新影响它。

对新手来说，可以把 PR 后审查写成这个流程：

```text
打开 PR 后：
1. 先看自动审查给出的高风险问题。
2. 把问题分成：必须修、可讨论、误报、以后再做。
3. 每次修复只改一个问题簇。
4. 修复后说明“这次只处理了什么”。
5. 再跑测试和链接/隐私检查。
6. 如果自动审查重复报旧问题，记录为已知项，不让它吞掉新增问题。
```

## 什么问题适合交给 Agent 审

Agent 审查擅长的是模式化、证据明确、可以和 diff 对照的问题。

| 适合 Agent 审 | 不适合只靠 Agent 审 |
|---|---|
| 是否改了无关文件 | 产品取舍是否正确 |
| 是否遗漏测试或链接更新 | 业务策略是否合理 |
| 是否有明显空指针、边界条件、重复逻辑 | 复杂安全威胁建模 |
| 是否出现密钥、本地路径、隐私词 | 法律、合规、合同判断 |
| README、目录、计数是否一致 | 是否应该上线影响真实用户 |

结论很简单：Agent 审查可以做第一轮和第二轮，但不能替代负责人判断。

## 新手可复制工作流

### 1. 开始前写 10 行 Spec

先用本仓库的 [Agent 编程前的 10 行 Spec 模板](agent-coding-10-line-spec-template.md) 写清：

- 目标。
- 不变量。
- 不做什么。
- 受影响文件。
- 验收命令。

没有 spec 的任务，后面审查时很难判断是否超范围。

### 2. 执行中加动作前门禁

让 Agent 在运行命令、改文件、联网、提交前先做风险分级。

重点不是每一步都问人，而是识别哪些动作必须停下来。

### 3. 写完后先看 diff

不要直接听 Agent 总结。先让它基于真实 `git diff` 做审查。

```text
请只基于当前 git diff 审查，不要基于你记忆里的计划。
如果 diff 中看不到证据，就不要声称已经完成。
```

### 4. 跑检查并保存证据

至少记录：

- 测试或构建命令。
- 格式检查。
- 链接检查。
- 隐私/密钥扫描。
- 未验证项。

文档型仓库也要做这些。不是只有代码才需要审查。

### 5. 再提交

提交信息要能说明本次变更的真实范围。

坏例子：

```text
update
```

好例子：

```text
Add agent code review workflow note
```

## 审查分级表

| 级别 | 含义 | 处理 |
|---|---|---|
| P0 | 会导致数据泄露、生产事故、错误发布、删除重要内容 | 立即停止，不提交 |
| P1 | 核心功能错误、验证缺失、严重越权修改 | 修完再提交 |
| P2 | 局部 bug、文档链接错误、计数不一致 | 一般本轮修 |
| P3 | 表达优化、结构微调、低风险建议 | 可记录到后续 |

Agent 审查如果只给“看起来没问题”，价值很低。好的审查必须指出问题级别、证据和建议处理方式。

## 常见坏用法

### 坏用法 1：让同一个 Agent 自证完成

同一个 Agent 写完代码后，通常会倾向于证明自己完成了。更稳的做法是让它切换到审查角色，并要求它只看 diff 和命令输出。

### 坏用法 2：没有任务边界就审查

如果一开始没写目标和不变量，审查时只能变成泛泛建议。审查必须有参照物。

### 坏用法 3：把自动审查当最终结论

自动审查可以发现很多问题，但它不知道你的真实业务优先级。合并、发布、删数据、改账号配置，仍然要负责人判断。

### 坏用法 4：忽略误报记录

自动审查会有误报。误报不能直接无视，应该记录为什么不处理，避免下次重复争论。

## 公开仓库维护者可以怎么用

像 Codex OpenSchool 这种知识库仓库，也适合用代码审查工作流。

每次新增内容前：

- 是否真的解决一个新手问题？
- 是否和已有笔记重复？
- 是否有一手来源？
- 是否写成了正文，而不是标题清单？

每次新增内容后：

- README 计数是否更新？
- `notes/README.md` 是否有目录项？
- 内部链接是否存在？
- 是否有敏感词或本地路径？
- 是否可以转成读者能直接使用的练习、提示词或判断表？

这就是把代码审查思路迁移到知识库维护。

## 最小执行清单

```text
开始前：
- [ ] 写 10 行 spec。
- [ ] 标出禁止动作。
- [ ] 标出必须人工确认的动作。

执行中：
- [ ] L2 以上动作先说明影响。
- [ ] 不读取密钥、账号、私有聊天、客户材料。
- [ ] 不做未授权发布、提交、删除。

提交前：
- [ ] 基于 git diff 审查。
- [ ] 跑测试或格式/链接检查。
- [ ] 扫描隐私和密钥。
- [ ] 写清未验证项。

PR 后：
- [ ] 先处理 P0/P1。
- [ ] 新 push 后聚焦新增 diff。
- [ ] 对误报写明原因。
- [ ] 合并前确认负责人判断。
```

## 不适合公开分享的材料

这类笔记可以讲方法，但不要公开：

- 真实公司 PR 截图。
- 内部安全审查规则。
- 包含密钥、账号、cookie、客户材料的日志。
- 未发布产品的代码 diff。
- 会暴露团队人员、仓库权限或生产环境的信息。

公开分享时，只保留抽象流程、脱敏示例和一手来源链接。
