# Agent 共享上下文与知识编译：别把长期协作塞进聊天记录

## 审计信息

- 整理时间：2026-06-12。
- 来源：本地 notebook 方法笔记“上下文压缩与文档外化”“知识编译工作流”“每日小结编译法”，以及 AIHot 发现的 DeLM 论文。
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 适合读者：已经让 Agent 连续做几轮任务，开始遇到“上下文太长、结论丢失、重复解释、换线程就断片”的新手。
- 目标：给新手一套可复制的共享上下文维护方法，把一次性聊天压缩成可验证、可复用、可继续协作的知识状态。
- 隐私处理：不包含本地路径、聊天记录、账号后台、客户材料、私有项目、密钥或未脱敏原始日志。

## 先说结论

Agent 长期协作的关键，不是把更多历史塞进聊天框，而是把重要进展压缩成外部文档。

一个稳定的共享上下文至少要回答五件事：

1. 当前目标是什么。
2. 已经确认了哪些事实。
3. 做过哪些决定，为什么。
4. 哪些假设被否定了。
5. 下一步任务队列是什么。

这和 AIHot 里 DeLM 论文的启发一致：多 Agent 协作不只需要“更多 Agent”，还需要一个可验证的共享上下文和任务队列。对个人用户来说，不一定要实现论文系统，但可以先学会维护一份轻量的 `shared-context.md`。

## 为什么聊天记录不是长期上下文

聊天记录很适合即时协作，但不适合长期复用。

| 聊天记录的问题 | 具体表现 | 更稳的做法 |
|---|---|---|
| 太长 | Agent 每次都要重新读一大段历史 | 压缩成当前状态摘要 |
| 噪音多 | 过程讨论、废稿、跑偏尝试混在一起 | 只保留已验证结论和仍有效约束 |
| 不可检索 | 想找一个决定要翻很多轮 | 把决定写进决策区 |
| 无法验收 | Agent 说“我记得”但没有证据 | 每条关键结论附来源或证据 |
| 换线程断片 | 新会话不知道之前怎么判断 | 新会话先读共享上下文文档 |

所以，“上下文管理”不是让 Agent 背更多东西，而是把协作状态外化出来。

## 最小共享上下文模板

先建一个很小的 Markdown 文件：

```text
# shared-context

## 当前目标
- ...

## 已验证事实
- [事实] ...
  证据：...

## 关键决定
- [决定] ...
  原因：...
  日期：...

## 被否定的假设
- [假设] ...
  为什么不成立：...

## 任务队列
- [ ] ...
- [ ] ...

## 待核验问题
- ...

## 变更日志
- YYYY-MM-DD：...
```

它不需要漂亮，先能用。新手最容易犯的错是想一上来搭完整知识库，结果每次整理都很重。共享上下文的第一目标是让下一轮 Agent 能接上。

## 什么内容可以写进共享上下文

共享上下文不是垃圾桶。只有对后续协作有用、已经核验或明确标注为待核验的信息，才值得写进去。

| 内容 | 是否写入 | 原因 |
|---|---|---|
| 当前目标和完成标准 | 写入 | 下一轮执行必须知道 |
| 已验证事实 | 写入 | 避免重复查证 |
| 决策和原因 | 写入 | 让后续修改有依据 |
| 被否定的方案 | 写入 | 避免反复走回头路 |
| 下一步任务 | 写入 | 形成任务队列 |
| 临时情绪、随口想法 | 不写入 | 噪音大，容易误导 |
| 原始聊天全文 | 不写入 | 太长，且可能有隐私 |
| 未脱敏账号、客户、后台内容 | 不写入 | 不应进入长期共享文档 |
| 未验证结论 | 可以写，但必须标 `待核验` | 防止冒充事实 |

一个简单门槛：如果这句话三天后还能帮助 Agent 少走弯路，就可以考虑写入；如果只是过程噪音，就留在聊天里。

## 五步知识编译流程

### 1. 先收集原始材料

原始材料可以是公开网页、任务结果、代码审查报告、每日小结、AI 输出、会议纪要或调试日志。它们先留在来源层，不直接变成结论。

不要让 Agent 把所有材料都改写成正式知识。先问：

```text
这批材料里，哪些会影响后续任务？
哪些只是一次性过程？
哪些还没有证据？
```

### 2. 提取可复用状态

从材料里只提取四类东西：

1. 已验证事实。
2. 决策和理由。
3. 可复用方法。
4. 后续任务。

示例：

```text
弱提取：今天试了三个方案。

强提取：
- 方案 A 已否定：原因是移动端按钮溢出无法通过现有样式修复。
- 方案 B 保留：只改表单容器宽度，不影响接口。
- 下一步：补 375px 和 768px 截图验收。
```

### 3. 压缩成短条目

共享上下文要短。每条最好是：

```text
结论 + 证据 + 对后续的影响
```

例子：

```text
- 表单布局问题只出现在 375px 宽度；桌面宽度未复现。
  证据：移动端截图和浏览器复验。
  影响：修复范围限定在移动端 CSS，不改接口。
```

不要写长篇复盘。长材料可以放在来源或输出里，共享上下文只保留可执行状态。

### 4. 入库前做验证

DeLM 论文里一个关键点是：共享上下文不应该是未经筛选的消息池，而应该只接收经过验证的紧凑更新。

个人实践里可以用三问替代复杂系统：

| 验证问题 | 通过标准 |
|---|---|
| 有证据吗？ | 有链接、截图、命令输出、diff、测试结果或人工确认 |
| 还有效吗？ | 没有被后续事实推翻 |
| 对后续有用吗？ | 能减少重复查证、避免返工或指导下一步 |

三问不过关，就不要写进“已验证事实”。可以放到“待核验问题”里。

### 5. 新会话先读共享上下文

每次开启新线程，不要把全部聊天历史贴过去。更稳的启动方式是：

```text
请先阅读 shared-context.md。
只把其中的“当前目标、已验证事实、关键决定、任务队列”作为本轮上下文。
不要把待核验问题当成事实。
如果你认为共享上下文过期，请先指出需要核验的条目，再继续。
```

这样 Agent 继承的是“当前协作状态”，不是一堆未整理的历史。

## 共享上下文和知识库的区别

共享上下文不是完整知识库。它更像当前任务的工作台。

| 层级 | 作用 | 生命周期 |
|---|---|---|
| `shared-context.md` | 当前任务状态、决策、队列、待核验问题 | 跟随任务变化，频繁更新 |
| `wiki/methods/` | 可跨任务复用的方法 | 长期有效，更新谨慎 |
| `wiki/decisions/` | 为什么选择某个方案 | 随项目演化更新 |
| `wiki/sources/` | 原始资料的来源摘要 | 用于追溯证据 |
| `outputs/` | 一次任务产出的报告、文章、代码审查 | 有价值时再回写正式知识 |

如果一条内容只服务当前任务，先留在共享上下文。反复出现三次以上，再考虑升级成方法页。

## 可复制提示词

### 把聊天记录压缩成共享上下文

```text
请把这段任务过程压缩成 shared-context.md。

要求：
1. 不保留完整聊天，只保留对后续任务有用的状态。
2. 把内容分成：当前目标、已验证事实、关键决定、被否定的假设、任务队列、待核验问题、变更日志。
3. 每条已验证事实必须有证据；没有证据的放进待核验问题。
4. 删除情绪、寒暄、重复解释和一次性过程。
5. 不写入账号、客户材料、聊天记录、密钥、本地路径或未脱敏截图。
```

### 让 Agent 在执行后更新共享上下文

```text
请根据本轮执行结果更新 shared-context.md。

只允许新增或修改这些部分：
- 已验证事实
- 关键决定
- 被否定的假设
- 任务队列
- 待核验问题
- 变更日志

要求：
1. 每个新增事实都写证据。
2. 如果证据不足，放到待核验问题。
3. 如果旧结论被推翻，移动到被否定的假设，并写明原因。
4. 不要把原始日志全文写入共享上下文。
5. 最后输出本次更新摘要。
```

### 开新线程时继承上下文

```text
请先读 shared-context.md，再继续这个任务。

执行规则：
1. 当前目标、已验证事实、关键决定、任务队列可以作为上下文。
2. 待核验问题不能当成事实。
3. 如果发现上下文过期、冲突或缺证据，先列出问题，不要直接执行。
4. 本轮完成后，请建议是否需要更新 shared-context.md。
```

## 一个低风险练习

选择一篇已经公开、已脱敏的学习笔记，做一次上下文外化练习：

1. 让 Agent 阅读笔记。
2. 让它输出 `shared-context.md`。
3. 检查“已验证事实”是否都有证据。
4. 把没有证据的移到“待核验问题”。
5. 开一个新任务，让 Agent 只读 `shared-context.md` 继续写一段总结。
6. 对比它是否比直接粘贴长聊天更稳定。

通过标准：

- 共享上下文不超过原始材料的 20%。
- 至少包含 3 条已验证事实。
- 至少包含 1 条被否定的假设或不做事项。
- 没有隐私、账号、本地路径或未脱敏材料。

## 常见失败模式

| 失败模式 | 表现 | 修正 |
|---|---|---|
| 把共享上下文写成流水账 | 每一步都记录，但没有结论 | 只保留结论、证据和后续影响 |
| 把待核验问题写成事实 | Agent 后续直接沿用错误结论 | 明确分区，未验证就不能进事实区 |
| 只写成功，不写失败 | 后续重复尝试已否定方案 | 保留“被否定的假设” |
| 共享上下文越来越长 | 读起来和聊天记录一样重 | 定期压缩，旧细节移到来源层 |
| 没有任务队列 | Agent 读完仍不知道做什么 | 每次更新都维护下一步任务 |
