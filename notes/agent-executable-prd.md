# AI 可执行 PRD：把一句话需求写成 Agent 能实现的产品规格

## 审计信息

- 整理时间：2026-06-12。
- 来源：AIHot 最近 7 天精选里的 `qiaomu-ai-prd`、Spec 驱动开发条目；结合 `qiaomu-ai-prd` 公开仓库、`warpdotdev/common-skills` 和本仓库的 Agent 任务执行经验整理。
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 适合读者：想让 Codex、Claude Code、Cursor、Replit Agent 等 AI 编程助手做产品或页面，但经常遇到“做出来不是我想要的”的新手。
- 目标：给新手一套能直接复制的 AI 可执行 PRD 结构，让 Agent 在实现前知道目标、边界、状态、约束和验收方式。
- 隐私处理：不包含本地路径、账号、客户材料、私有仓库、真实业务数据、密钥或未发布项目；不复制外部 skill 的完整提示词。

## 先说结论

普通 PRD 经常写给人看，AI 可执行 PRD 要写给“会自己补细节的 Agent”看。

区别不在于文档更长，而在于它必须减少 Agent 的猜测空间：

```text
普通需求：做一个学习网站。
AI 可执行 PRD：谁学习、学习什么、核心循环是什么、哪些状态必须有、哪些地方不能自由发挥、怎么验收。
```

AIHot 里出现的 `qiaomu-ai-prd` 很值得拆解。它的价值不是“又多一个 PRD 模板”，而是把一句话想法转成包含速读卡、约束层、布局、状态、数据模型、优先级、性能指标和验收剧本的开发输入。

新手不一定要马上安装这个 skill，但应该学会它背后的原则：**让 Agent 看得懂、做得出、验得了。**

## AI 可执行 PRD 和普通 PRD 的区别

| 维度 | 普通 PRD 常见写法 | AI 可执行 PRD 应该写法 |
|---|---|---|
| 目标 | 描述产品愿景 | 描述最小可实现闭环 |
| 用户 | 写几类用户画像 | 说明本次 P0 先服务谁 |
| 功能 | 列功能清单 | 写用户行为、系统响应和边界状态 |
| 约束 | 写“体验好”“易用” | 拆成硬约束、推荐默认、发挥空间 |
| 交互 | 描述页面大概样子 | 写布局、状态、失败路径和空状态 |
| 技术 | 泛泛说技术栈 | 只写会影响行为、数据和验收的技术要求 |
| 验收 | 产品经理人工看 | 给 Agent 可执行或可人工复核的验收剧本 |

真正的关键是：Agent 不怕文档短，怕文档里全是愿望。

## 一份 AI 可执行 PRD 的最小结构

新手可以先用这 12 块，不需要一上来写几十页。

```text
1. AI 速读卡
2. 产品一句话
3. 目标用户和核心场景
4. 当前问题
5. 核心循环
6. 硬约束 / 推荐默认 / 发挥空间
7. 页面或模块结构
8. 关键状态和失败路径
9. 数据模型或内容结构
10. P0-P2 优先级
11. 可量化指标
12. 给实现 Agent 的交接和验收剧本
```

如果只能写 5 块，优先保留：

```text
AI 速读卡
核心循环
硬约束 / 推荐默认 / 发挥空间
关键状态和失败路径
验收剧本
```

## 第一块：AI 速读卡

速读卡是给实现 Agent 的 10 行以内上下文。它要让 Agent 在开始改代码前就知道“这东西到底是什么”。

模板：

```text
产品名称：
产品一句话：
目标用户：
核心问题：
核心循环：
P0 必须完成：
硬约束：
推荐默认：
发挥空间：
验收方式：
```

示例：

```text
产品名称：Agent Notes
产品一句话：一个帮助新手把公开网页整理成 Markdown 学习笔记的小工具。
目标用户：刚开始用 AI Agent 学习资料的新手。
核心问题：他们会让 AI 摘要网页，但不会保存来源、边界和行动项。
核心循环：输入公开 URL -> 生成结构化笔记 -> 自检来源和风险 -> 保存 Markdown。
P0 必须完成：一篇笔记能包含来源、核心观点、行动项、待查证和自检结论。
硬约束：不处理登录页、付费墙、私有文档或聊天记录。
推荐默认：输出中文 Markdown；保留原始 URL；使用简洁标题。
发挥空间：可以优化笔记排版和标签建议。
验收方式：用一篇公开网页跑通，检查 Markdown 字段完整且无隐私内容。
```

## 第二块：核心循环，不要只列功能

Agent 最容易误解“功能清单”。例如：

```text
- 支持登录
- 支持收藏
- 支持搜索
- 支持分享
```

这对 Agent 来说不够，因为它不知道用户为什么要点这些按钮，也不知道最小闭环是什么。

改成核心循环：

```text
用户打开页面 -> 输入一个公开资料链接 -> 系统生成一篇学习笔记 -> 用户检查来源和不确定项 -> 保存到知识库。
```

然后再问：

| 问题 | 示例答案 |
|---|---|
| 用户第一步做什么？ | 粘贴 URL |
| 系统第一次返回什么？ | 页面标题、摘要、可整理性判断 |
| 用户如何确认？ | 点击“生成笔记”或编辑提示 |
| 成功后看到什么？ | Markdown 预览和保存按钮 |
| 失败时怎么办？ | 提示页面不可访问，建议换公开资料 |

核心循环写清楚后，功能优先级自然会收敛。

## 第三块：硬约束、推荐默认、发挥空间

这是 AI 可执行 PRD 里最重要的一块。

很多人写需求时只有“必须做”和“随便你”。但 Agent 需要更细的自由度：

| 类型 | 作用 | 例子 |
|---|---|---|
| 硬约束 | 不能违反，违反就算失败 | 不读取私有文件；不自动发布；P0 必须完成核心闭环 |
| 推荐默认 | 没有更好理由时按这个做 | Markdown 输出；移动端优先；本地保存优先 |
| 发挥空间 | 允许 Agent 优化和加分 | 空状态文案、动效、卡片视觉、快捷入口 |

这样写的好处是：既不把 Agent 限制死，也不会让它在关键地方乱发挥。

反例：

```text
页面要高级、丰富、智能。
```

改成：

```text
硬约束：首屏必须出现输入框、示例链接、生成按钮和隐私边界说明。
推荐默认：桌面端左右布局，左侧输入，右侧预览；移动端上下布局。
发挥空间：可以增加最近生成记录、标签建议和保存成功动效，但不能影响 P0。
```

## 第四块：状态和失败路径

新手写 PRD 最容易只写 happy path。Agent 也会顺着最乐观路径实现，结果一遇到空数据、错误输入、网络失败就崩。

至少写 6 类状态：

| 状态 | PRD 里要写什么 |
|---|---|
| 默认态 | 页面刚打开时看到什么 |
| 输入态 | 用户输入时有什么反馈 |
| 加载态 | 正在处理时能否取消、是否有进度 |
| 成功态 | 成功后显示什么、能做什么下一步 |
| 空状态 | 没有内容、没有结果、首次使用时怎么办 |
| 错误态 | URL 无法访问、格式错误、权限不足时怎么办 |

示例：

```text
错误态：
- 如果 URL 需要登录，系统不尝试绕过，提示“这个页面可能不是公开资料，请换一个公开链接”。
- 如果页面读取成功但正文太短，提示“正文不足，可能不是文章页”，并允许用户手动粘贴正文。
- 如果生成笔记失败，保留用户输入，不清空页面。
```

## 第五块：数据模型不是程序员专用

AI 可执行 PRD 里的数据模型不是为了提前写代码，而是为了让 Agent 知道“信息结构是什么”。

轻量模板：

```json
{
  "version": "1.0",
  "source": {
    "title": "资料标题",
    "url": "公开链接",
    "access_checked_at": "整理日期"
  },
  "note": {
    "audience": "适合谁",
    "problem": "解决什么问题",
    "key_points": ["核心观点"],
    "actions": ["可执行动作"],
    "uncertainties": ["待查证"]
  },
  "audit": {
    "has_source": true,
    "privacy_risk": "none",
    "copyright_risk": "low"
  }
}
```

写数据模型时要避免两个极端：

- 不写结构，让 Agent 自己猜字段。
- 写过度技术化的数据库 schema，把产品行为锁死。

原则是：只写会影响输入、输出、验收和后续维护的字段。

## 第六块：优先级用 P0-P2，不要一口吃完

把所有想法都写进 P0，Agent 会做出一个大而散的半成品。

推荐分法：

| 优先级 | 含义 | 判断标准 |
|---|---|---|
| P0 | 没有它，核心循环不成立 | 第一次使用必须完成 |
| P1 | 有它，体验明显更完整 | 可以在 P0 后继续做 |
| P2 | 加分项或可选增强 | 做不做不影响主流程 |

示例：

```text
P0：
- 输入公开 URL。
- 生成结构化 Markdown 笔记。
- 标出来源和不确定项。
- 保存或复制 Markdown。

P1：
- 支持手动粘贴正文。
- 支持标签建议。
- 支持生成前预览资料标题。

P2：
- 最近笔记历史。
- 多主题模板。
- 分享卡片。
```

P0 的判断标准很简单：没有它，用户是否还能完成核心任务？不能，就放 P0；能，就往后放。

## 第七块：验收剧本要写给 Agent

不要只写“验收通过”。要给 Agent 一个可以执行或复核的剧本。

模板：

```text
验收剧本 1：正常路径
1. 使用一个公开文章 URL。
2. 点击生成。
3. 看到 Markdown 预览。
4. Markdown 至少包含：来源、适合谁、核心观点、行动项、待查证、自检。
5. 复制结果后格式不丢失。

验收剧本 2：非公开页面
1. 输入一个需要登录的链接。
2. 系统不尝试绕过限制。
3. 页面提示用户换公开链接或手动粘贴正文。

验收剧本 3：隐私边界
1. 输入提示中包含“微信聊天记录”“客户资料”“后台截图”等词。
2. 系统提示需要脱敏，不直接处理。
```

对代码项目，可以再补：

```text
验证命令：
- npm test
- npm run lint
- 手动打开本地页面，检查桌面和移动端布局
```

如果某条无法自动验证，就明确写“人工检查点”，不要伪装成自动测试。

## 可复制提示词：把一句话变成 AI 可执行 PRD

```text
你是一个严谨的 AI 产品规格助手。
请把我的一句话产品想法改写成 AI 可执行 PRD。

原始想法：
[粘贴想法]

要求：
1. 不要直接实现，也不要写代码。
2. 如果信息不足，优先做保守假设；只有会影响平台、数据安全、法律风险、预算或核心范围的问题才向我追问。
3. 输出必须让 AI 编程助手可以据此实现和验收。
4. 不要写空泛词，例如“高级、智能、好用”，必须转成可观察行为。
5. 不要编造当前竞品、API、包版本或平台能力；不确定就写“待核验”。

输出结构：

- 产品一句话：

- 发挥空间：
- 验收方式：

## 用户和场景

## 核心循环

## 功能优先级
| 优先级 | 功能 | 为什么 |

## 页面或模块结构

至少写 3 条：正常路径、失败路径、隐私/权限边界。
```

## 可复制提示词：审计 PRD 是否适合交给 Agent

```text
请审计下面这份 PRD 是否适合交给 AI 编程助手实现。

PRD：
[粘贴 PRD]

请按表格输出：
| 检查项 | 结论 | 证据 | 修改建议 |

检查项：
1. 是否有明确目标用户和核心循环。
2. 是否把约束拆成硬约束、推荐默认、发挥空间。
3. 是否覆盖默认态、加载态、成功态、空状态、错误态。
4. 是否有 P0-P2 优先级，且 P0 不过大。
5. 是否有数据模型或内容结构。
6. 是否有可执行或可人工复核的验收剧本。
7. 是否有隐私、权限、版权或安全边界。
8. 是否存在空泛词、未核事实或无法验证的承诺。

最后给出判断：
- 可以交给 Agent 实现
- 需要补充后再实现
- 不建议实现，原因是...
```

## 什么时候直接用 qiaomu-ai-prd

`qiaomu-ai-prd` 是向阳乔木开源的 AI PRD skill，适合这些场景：

- 你只有一句产品想法，但希望快速得到一份更完整的 PRD。
- 你准备把需求交给 AI 编程助手实现。
- 你希望 PRD 包含约束层、模块、数据结构、优先级和验收剧本。
- 你接受 skill 帮你做一部分产品判断，而不是只填空。

不建议用它的场景：

- 你只是改一个很小的 bug。
- 你已经有详细产品稿，只需要转成 issue。
- 需求涉及真实用户数据、支付、医疗、法律、金融等高风险场景，还没有人工审核。
- 你需要严格遵守公司内部 PRD 模板，不能让 Agent 自由重组结构。

公开仓库入口：[joeseesun/qiaomu-ai-prd](https://github.com/joeseesun/qiaomu-ai-prd)。

## 和 Spec 驱动开发的关系

这篇笔记和 [Agent 编程前的 10 行 Spec 模板](agent-coding-10-line-spec-template.md) 是递进关系：

| 任务规模 | 推荐做法 |
|---|---|
| 30 分钟以内的小任务 | 10 行 Spec |
| 一个页面或小工具 | AI 可执行 PRD |
| 多模块功能 | `PRODUCT.md` + `TECH.md` |
| 团队长期项目 | Spec workflow、实现审查、trace 和回放 |

`warpdotdev/common-skills` 的 Spec workflow 强调先写 `PRODUCT.md`，再写 `TECH.md`，最后核对实现是否符合 spec。这个思路和 AI 可执行 PRD 是一致的：先让行为和验收变清楚，再让 Agent 动手。

## 发布前自检清单

把 PRD 交给 Agent 前，逐条检查：

| 检查项 | 通过标准 |
|---|---|
| 读者 | 明确谁使用，不写泛泛“用户” |
| 核心循环 | 能用一句话说明从开始到完成 |
| P0 | 只包含核心闭环，不把所有愿望塞进去 |
| 约束 | 分清不能违反、推荐默认和可发挥空间 |
| 状态 | 至少覆盖默认、加载、成功、空、错误 |
| 数据 | 有输入输出结构，不让 Agent 猜字段 |
| 验收 | 有正常路径、失败路径和权限/隐私边界 |
| 来源 | 涉及外部事实时有链接或标待核验 |
| 隐私 | 不包含账号、聊天记录、客户材料、密钥或本地路径 |

如果这张表有 3 项以上不通过，先改 PRD，不要急着实现。
