# Agent 自主权限分级清单：什么时候放手，什么时候必须停下来

## 审计信息

- 来源：本地 Codex / 自动化浏览 / 文件操作 / GitHub 发布经验；结合 AIHot 2026-06-12 发现的 Cursor Auto-review 动态、Cursor 官方博客、OpenAI Agents SDK guardrails 文档和 Anthropic agent 工程文章整理。
- 作者：本地整理；公开版由 Codex 脱敏改写。
- 适合读者：已经开始让 Agent 调工具、改文件、浏览网页、生成草稿，但不确定哪些动作能自动执行的新手。
- 目标：给出一套可复制的权限分级表，让新手把 Agent 的自主性从“全放开 / 每步问”改成按风险分层。
- 隐私处理：不包含本地路径、账号、聊天记录、后台数据、私有 skill 源码、密钥或未发布项目。

## 为什么要做权限分级

新手用 Agent 最常见的两种极端：

1. 什么都让 Agent 自己做，直到它误删文件、泄露上下文、发错内容。
2. 每一步都要求确认，最后人被确认弹窗淹没，反而不认真看。

Cursor 的 Auto-review 文章给了一个很好的思路：Agent 自主性不是开关，而是刻度。低风险动作应该顺畅执行，高风险动作应该被拦下、降级或要求更明确的人工确认。

OpenAI Agents SDK 的 guardrails 文档也提醒：检查可以放在不同位置，输入、输出、工具调用各有边界。对新手来说，不需要一开始就实现复杂分类器，但应该先有一张清楚的权限分级清单。

## 一张表先用起来

| 等级 | 动作类型 | 例子 | 默认处理 | 需要留下的证据 |
|---|---|---|---|---|
| L0 | 只读公开资料 | 读取公开网页、官方文档、开源 README | 可自动执行 | 来源链接、读取时间、摘要 |
| L1 | 生成草稿 | 写 Markdown 初稿、整理表格、生成提示词、改写公开笔记 | 可自动执行，但要自检 | diff、草稿文件、引用来源 |
| L2 | 修改本地非敏感文件 | 编辑仓库文档、调整代码、运行格式化、生成 PPT 草稿 | 先说明计划，执行后验证 | git diff、命令输出、检查结果 |
| L3 | 有外部副作用 | push 代码、创建 issue、发帖、发邮件、提交表单、调用付费 API | 必须人工确认 | 操作对象、影响范围、回滚方式 |
| L4 | 高危动作 | 读取密钥、处理生产数据、付款、下单、绕过验证码、修改账号安全设置 | 默认禁止 | 只有用户明确授权且有安全环境时才讨论 |

新手默认规则：

```text
L0-L1：可以自动做，但必须保留来源和自检。
L2：可以做，但先说计划，做完必须验证。
L3：先生成草稿或计划，人工确认后再执行。
L4：默认不做，除非用户明确授权、环境安全、风险可控。
```

## 怎么判断一个动作属于哪一级

不要只看命令名字，要看“错了会怎样”。

| 判断问题 | 如果答案是“是” | 风险倾向 |
|---|---|---|
| 是否会修改已有文件？ | 是 | 至少 L2 |
| 是否会影响远程仓库、账号、外部平台或其他人？ | 是 | 至少 L3 |
| 是否可能读取密钥、cookie、token、私有聊天、客户材料？ | 是 | L4 |
| 是否会产生费用、付款、下单或发送消息？ | 是 | L3-L4 |
| 是否绕过登录、验证码、付费墙或平台限制？ | 是 | L4 |
| 是否无法可靠回滚？ | 是 | 升一级处理 |
| 是否只是读取公开资料并输出草稿？ | 是 | 通常 L0-L1 |

同一个动作在不同上下文里风险不同。

例如 `python script.py`：

- 如果脚本只把公开 CSV 转成 Markdown，可能是 L2。
- 如果脚本会删除文件、访问生产数据库或上传结果，就是 L3-L4。
- 如果不知道脚本做什么，先读脚本，再决定等级。

## 工具调用前检查表

让 Agent 调工具前，先让它回答这 6 个问题：

```text
1. 你准备做的下一个动作是什么？
2. 这个动作属于 L0 / L1 / L2 / L3 / L4 哪一级？
3. 为什么是这个等级？
4. 最坏情况是什么？
5. 你会如何验证结果？
6. 如果风险过高，有没有更安全的替代动作？
```

如果 Agent 回答不清楚，就不要让它继续执行。

## 给 Agent 的可复制提示词

把这段加到涉及工具调用、文件操作、浏览器自动化或发布动作的任务里：

```text
在执行任何工具调用或外部动作前，请先做权限分级。

权限等级：
- L0：只读公开资料，可自动执行。
- L1：生成草稿或新建非敏感内容，可自动执行，但要自检。
- L2：修改本地非敏感文件或运行本地命令，先说明计划，执行后验证。
- L3：影响远程仓库、账号、外部平台、他人可见内容或付费资源，必须等待我明确确认。
- L4：读取密钥、处理生产数据、付款、下单、绕过登录/验证码/付费墙、修改账号安全设置，默认禁止。

每次行动前输出：
1. 下一步动作。
2. 权限等级。
3. 风险理由。
4. 验证方式。
5. 是否需要我确认。

如果不确定等级，按更高风险处理。
```

## 三个护栏应该放在哪里

OpenAI Agents SDK 文档把 guardrails 放在输入、输出和工具调用等不同位置。新手可以先用人话理解：

| 护栏位置 | 解决什么问题 | 新手例子 |
|---|---|---|
| 输入前 | 判断任务能不能接 | 用户给了聊天记录、密钥、后台截图，先拒绝或要求脱敏 |
| 工具前 | 判断下一步动作能不能做 | 准备运行命令、提交表单、push 代码前做权限分级 |
| 输出后 | 判断结果能不能交付 | 检查是否有幻觉、无来源判断、隐私泄露、版权风险 |

不要只在最后检查。很多风险发生在工具调用之前。

## 最小可行实现：不写分类器，也能先变安全

Cursor 使用分类器 Agent 做动态审查，这是产品级实现。个人新手不需要一开始就做这么复杂，可以先用三个轻量动作：

1. 在提示词里写明权限等级。
2. 在每次工具调用前要求 Agent 自报等级。
3. 在任务结束后检查 diff、来源、输出和未确认动作。

这已经能挡住大部分新手风险。

如果以后要做成更稳定的 skill 或脚本，再考虑：

- 给常见动作写 allowlist / denylist。
- 对命令、URL、文件路径做规则检查。
- 对 L3-L4 动作强制人工确认。
- 保存每次工具调用和判断结果，方便复盘。

## 常见场景怎么分级

| 场景 | 推荐等级 | 处理方式 |
|---|---|---|
| 整理一篇公开网页成笔记 | L0-L1 | 自动执行，保留来源，输出自检 |
| 新建一篇 Markdown 教程 | L1 | 自动生成，人工审稿后发布 |
| 修改仓库 README | L2 | 先说明修改范围，做完看 diff |
| 运行测试或链接检查 | L2 | 可执行，保留命令输出 |
| commit 并 push 到 GitHub | L3 | 明确列出 staged 文件和 commit 信息，再推送 |
| 自动发布到小红书/公众号/X | L3 | 只生成草稿，发布前人工确认 |
| 读取浏览器登录态页面 | L3-L4 | 先确认页面性质，不处理账号后台或隐私数据 |
| 读取 `.env`、token、cookie | L4 | 默认禁止 |
| 处理客户资料、真实聊天、订单数据 | L4 | 默认不进入公开仓库，先脱敏和授权 |
| 绕过验证码、付费墙、平台限制 | L4 | 默认禁止 |

## 失败案例：为什么“每步确认”不够

如果 Agent 每次都问：

```text
我要运行命令，可以吗？
我要继续处理，可以吗？
我要写文件，可以吗？
```

用户很快会开始机械点同意。

更好的确认应该包含：

```text
我要修改 README.md 和 notes/README.md，把新笔记接入目录。
风险等级：L2，因为只修改本地仓库文档，不影响远程。
验证方式：查看 git diff、跑内链检查、确认没有隐私路径。
不涉及：账号、密钥、发布、付款、外部系统。
是否继续？
```

确认不是为了打断流程，而是让用户知道“具体要发生什么、错了怎么发现”。

## 自检清单

每次让 Agent 做一个真实任务前，用这张表：

| 检查项 | 通过标准 |
|---|---|
| 等级 | 每个外部动作都标了 L0-L4 |
| 范围 | 说清楚会读什么、改什么、不会碰什么 |
| 证据 | 输出里有来源、diff、命令结果或截图 |
| 人工确认 | L3 动作没有自动执行 |
| 禁止项 | L4 动作默认拒绝或要求脱敏/授权 |
| 回滚 | L2-L3 动作知道怎么撤回或修正 |
| 发布 | 对外发布只生成草稿，不自动发 |

## 一句话收尾

Agent 越能做事，越需要权限分级。新手先别追“全自动”，先做到：低风险自动跑，中风险可验证，高风险先确认，禁止项不碰。
