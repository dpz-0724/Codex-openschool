# 企业 Agent 数据治理与自助分析落地手册

这篇笔记讨论一个很实际的问题：企业把 Agent 接到数据系统以后，为什么不能让它“自己理解业务、自己查数据、自己解释结果”。

适合读者：

- 想做企业数据分析 Agent、BI Agent、自助分析助手的人。
- 正在把 SQL、指标平台、知识库和 Agent 工作流接到一起的人。
- 需要向客户解释“为什么 Agent 落地前要先做数据治理”的人。

## 一句话结论

企业 Agent 的数据治理，就是把“怎么做、词是什么意思、数据从哪里来、指标怎么算、结果怎么验”提前标准化，避免 Agent 自己猜。

企业场景里，Agent 最危险的错误不是“不会写 SQL”，而是：

- 用错数据源。
- 误解业务词。
- 算错指标口径。
- 漏掉默认过滤条件。
- 给出看起来合理但无法复核的解释。

## 五类核心资产

### 1. Skill / SOP：规定怎么做

Skill 可以理解为写给 Agent 的岗位 SOP。

比如用户问：

```text
为什么本月新用户下降了？
```

Agent 不应该直接开始查表，而应该按流程：

```text
1. 确认“本月”是自然月还是最近 30 天。
2. 查公司标准的新用户定义。
3. 使用官方指标和官方数据模型。
4. 与上月、去年同期对比。
5. 按渠道、地区、产品版本拆分。
6. 检查数据同步、埋点、口径变化。
7. 输出结论、证据、口径、来源和置信度。
```

`Skill / SOP` 解决的是：Agent 遇到这类问题应该按什么流程工作。

### 2. 业务定义：规定词是什么意思

企业里的词很容易被不同团队理解成不同东西。

例如“新用户”：

```text
市场视角：新注册账号。
销售视角：CRM 新建客户。
财务视角：第一次付款客户。
产品视角：第一次完成核心行为的用户。
```

如果不提前定义，Agent 会自己猜。

一个可执行的业务定义应该写成：

```text
新用户 = 在指定时间范围内首次完成账号注册的外部用户
不包含 = 内部员工账号、测试账号、欺诈账号、重复账号
时间字段 = signup_date
唯一 ID = user_id
```

`业务定义` 解决的是：用户说的业务词到底指什么。

### 3. 数据底座：规定数据从哪里来

企业里有很多系统，但不是每个系统都适合让 Agent 直接查询：

```text
注册系统
交易系统
CRM
广告投放系统
产品行为日志
客服系统
财务系统
```

Agent 需要知道哪些是官方可信数据，哪些只是原始数据，哪些已经清洗去重，哪些不能直接用于分析。

例如：

```text
用户相关问题，优先查用户标准数据集。
收入相关问题，优先查收入标准数据集。
渠道相关问题，优先查渠道归因标准数据集。
客服相关问题，优先查客服工单标准数据集。
```

`数据底座` 解决的是：Agent 可以信任哪些数据。

### 4. Canonical Model：规定某类问题的官方数据模型

Canonical Model 可以理解为某个业务域的官方标准视图。

比如分析“新用户 7 日激活率”，不要让 Agent 去原始事件表里猜：

```text
哪个表代表新用户？
哪个事件代表激活？
要不要排除测试账号？
渠道用哪个字段？
7 天是自然日还是 168 小时？
```

应该提前做出官方模型：

```text
模型：user_activation_canonical
一行：一个新注册用户
新用户：完成注册的外部用户
激活：注册后 7 天内完成核心行为
默认排除：内部账号、测试账号、欺诈账号
可拆分维度：月份、渠道、地区、产品版本
```

`Canonical Model` 解决的是：这类业务问题应该查哪个官方模型。

### 5. 指标手册 / 语义层：规定指标怎么算

业务定义解释词，指标手册解释数。

例如：

```text
新用户数 = 某时间范围内 signup_date 落入该范围的 user_id 去重数量
7 日激活率 = 注册后 7 天内完成核心行为的新用户数 / 新用户总数
```

一个可用的指标定义至少包含：

```text
指标名称
业务含义
使用模型
分子
分母
默认过滤条件
时间窗口
可拆分维度
负责人
数据更新频率
```

`指标手册 / 语义层` 解决的是：Agent 如何把业务语言映射成标准计算。

## 五类资产如何协同

可以用这条链路理解：

```text
用户提问
  ↓
Skill / SOP 判断该怎么处理
  ↓
业务定义解释词是什么意思
  ↓
语义层解释指标怎么算
  ↓
Canonical Model 决定查哪个官方模型
  ↓
业务知识库补充背景、变化和例外
  ↓
验证机制检查结果是否可信
  ↓
输出结论、证据、口径和置信度
```

压缩成一句：

```text
Canonical Model 负责查哪里。
业务定义负责词是什么意思。
语义层负责怎么算。
Skill 负责怎么做。
Validation 负责有没有做对。
```

## 业务知识库和业务定义的区别

这两个经常被混在一起。

业务定义是标准词典：

```text
渠道 = 用户首次归因的获客渠道
```

业务知识库是分析背景：

```text
某个时间点之前，渠道字段来自注册来源。
某个时间点之后，渠道字段来自归因系统。
某次活动可能影响了渠道质量。
某个地区的数据同步存在延迟。
```

区别是：

```text
业务定义：这个词是什么意思。
指标手册：这个数怎么算。
业务知识库：分析这个数时要知道哪些背景、变化、例外和坑。
```

## 最小可落地版本

不要一开始就做全公司级数据治理。先选一个高频场景，做最小闭环。

例如先做“用户激活分析 Agent”：

```text
1. 一个官方数据模型
   user_activation_canonical

2. 一组业务定义
   新用户、激活、企业版、渠道

3. 一组指标
   新用户数、7 日激活率、按渠道激活率

4. 一份业务知识库
   字段变化、活动影响、产品版本变化、地区数据延迟

5. 一个 Skill / SOP
   遇到激活问题时怎么分析

6. 30-50 道验收题
   测试 Agent 是否使用正确数据、正确口径和正确流程
```

最小闭环是：

```text
一个场景
一个官方数据模型
一套业务定义
一套指标口径
一个 Agent SOP
一组验收题
```

## 给客户解释时怎么说

可以这样讲：

```text
我们不是简单把 Agent 接到数据库上，让它自己查。
那样很容易出现口径混乱、数据源误用、结果看起来合理但其实错误的问题。

真正要建设的是 AI 可用的数据治理体系：
先明确哪些数据可信，
再统一业务术语和指标口径，
然后把这些规则写成 Agent 的 SOP，
最后用测试题持续验证 Agent 有没有按规则做。
```

## 交付清单

如果要把这个方案交付给一个企业客户，至少要有这些东西：

```text
1. 场景清单：先做哪些高频分析问题。
2. 业务术语表：关键业务词的标准定义。
3. 指标口径表：核心指标的计算规则。
4. 官方数据模型列表：每类问题默认查哪里。
5. Agent SOP：Agent 遇到某类问题时怎么处理。
6. 验收题库：用来测试 Agent 是否做对。
7. 审计日志规范：记录输入、查询、口径、输出和人工确认。
```

## 最终判断

企业 Agent 落地的核心，不是“把数据库开放给模型”，而是把企业知识和数据治理做成 Agent 可执行的规则体系。

真正要问的是：

```text
它会不会用公司的正确数据？
它懂不懂公司的业务词？
它会不会按公司的指标口径计算？
它能不能证明自己没有乱猜？
```
