# Codex Goal 指令生成提示词

## 审计信息

- 整理时间：2026-06-12。
- 来源：AIHot 发现的 [qiaomu-goal-meta-skill](https://github.com/joeseesun/qiaomu-goal-meta-skill) 启发；公开版为本地重写模板。
- 原作者：向阳乔木 / joeseesun。
- 适合场景：把一句模糊任务改写成 Codex 可执行、可验证、有边界的 `/goal`。
- 隐私边界：不要在提示词里粘贴账号、密钥、客户材料、真实聊天记录、未脱敏截图或生产数据。

## 模板

```text
请把下面这个模糊需求改写成一条适合 Codex 执行的 /goal 指令。

原始需求：
[粘贴你的需求]

背景：
- 当前项目/仓库/资料状态：[一句话说明，没有就写“未知”]
- 我希望优先得到的产物：[文件、页面、功能、笔记、报告、脚本等]
- 我不希望触碰的内容：[账号、生产、支付、隐私、版权素材、无关目录等]

请输出：
1. 推荐执行版：一条可直接复制的中文 /goal。
2. 默认选择理由：用 1-2 句话说明为什么这样收窄范围。
3. 可选调整：列 3-5 个关键选项，让我可以用 A/B/C 快速选择。
4. 风险提醒：哪些情况必须暂停人工确认。

/goal 必须包含：
- 目标结果：最后要交付什么。
- 验证：用什么命令、截图、文件、链接、浏览器流程或人工检查证明完成。
- 约束：明确不做什么。
- 边界：允许读取或修改哪里。
- 迭代策略：先做哪一版，最多几轮改进。
- 完成条件：什么同时满足才算结束。
- 暂停条件：遇到账号、密钥、付款、生产、隐私、版权、合规或范围变化时如何停。

要求：
- 不要输出占位符。
- 不要写“随便改”“一直尝试”“确认可用”这种无法验收的话。
- 不要默认允许上线、付款、登录真实账号、读取密钥或处理未脱敏数据。
- 如果领域陌生或高风险，默认写成“先发现、再实现”的 goal。
```

## 用法建议

先用这个模板把任务写成 `/goal`，再把生成的 `/goal` 交给 Codex 执行。这样可以把“想让 AI 做什么”和“AI 具体怎么做”分开，减少中途跑偏。

如果任务很小，可以只保留这 5 行：

```text
/goal [目标结果]
验证：[证据]
约束：[不做什么]
边界：[能改哪里]
暂停条件：[何时问人]
```
