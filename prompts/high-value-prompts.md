# 高价值提示词模板精选

这些模板不是为了凑数量，而是为了覆盖 Codex OpenSchool 最常见、最有复用价值的任务：让 Agent 做事可控、让资料研究有证据、让 PPT 和视频工作流可交付。

## 1. Codex 任务拆解与验收模板

用途：把一个模糊需求变成 Codex 可执行任务，避免它直接开干。

```text
你是一个严谨的 Codex 执行代理。请先把我的任务拆解清楚，再开始执行。

任务：{任务描述}
上下文：{已有资料、文件、限制}
最终交付物：{文件、报告、代码、PPT、脚本等}

先输出：
1. 你理解的目标。
2. 完成标准：什么算真的完成。
3. 需要读取或确认的资料。
4. 风险点：隐私、版权、凭据、误操作、过期信息。
5. 执行步骤。
6. 验收方式：你会如何证明结果可靠。

约束：
- 不要为了速度跳过验收。
- 不确定的信息先标注，不要编造。
- 涉及真实账号、客户材料、聊天记录、密钥、本机路径时，先做脱敏判断。
```

## 2. 来源驱动研究简报模板

用途：做 AI 工具、GitHub 项目、技术趋势、竞品或论文调研，强制每个关键结论可追溯。

```text
请围绕 {研究主题} 做一份来源驱动研究简报。

范围：
- 受众：{给谁看}
- 时间范围：{是否需要最新信息}
- 优先来源：官方文档、GitHub README、论文、发布公告、源码、权威文章
- 排除来源：营销软文、二手搬运、无来源截图

输出：
1. 5 条以内核心结论。
2. 证据表：结论、来源链接、发布时间或提交时间、可信度。
3. 哪些信息是事实，哪些是你的推断。
4. 对 {我的场景} 的具体建议。
5. 还需要人工确认的问题。

要求：
- 不用单一搜索结果证明结论。
- 官方和源码优先。
- 如果资料可能过期，明确写复核日期。
```

## 3. Agent 工作流选择模板

用途：判断一个任务到底该用提示词、skill、脚本、MCP，还是自主 Agent，避免过度 Agent 化。

```text
请判断这个任务最适合做成什么形态：

任务：{任务描述}
输入：{输入类型}
输出：{输出类型}
是否重复出现：{一次性 / 偶尔 / 高频}
是否需要调用工具：{需要 / 不需要}
是否会产生副作用：{会 / 不会 / 不确定}
失败成本：{低 / 中 / 高}

候选形态：
- 一次性提示词
- 可复用 prompt 模板
- Codex skill
- 脚本 / CLI
- MCP 工具
- 多 Agent 工作流
- 暂不自动化

请输出：
1. 推荐形态。
2. 为什么不是其他形态。
3. 最小可行版本。
4. 权限和安全边界。
5. 未来什么时候升级到更复杂形态。
```

## 4. PR 代码审查模板

用途：让 AI 做第一轮审查，重点找确定性问题，而不是泛泛评价。

```text
你是资深代码审查员。请审查下面的 diff：{diff 或文件摘要}。

请优先找：
1. 会导致错误结果、数据损坏、安全漏洞或线上故障的问题。
2. 边界条件和异常路径。
3. 缺失的关键测试。
4. 与现有代码风格或架构不一致的地方。

每个发现必须包含：
- 严重程度：P0/P1/P2/P3
- 文件和行号，如可判断
- 问题是什么
- 为什么会发生
- 建议修复
- 建议测试

如果没有确定性问题，请明确说“未发现确定性问题”，并列出剩余风险。

不要输出泛泛的“建议提高可读性”，除非它会导致真实维护风险。
```

## 5. CI 失败诊断模板

用途：读失败日志，按假设驱动定位问题。

```text
请诊断这次 CI 失败。

失败日志：{日志}
最近变更：{变更摘要}
相关环境：{语言、框架、依赖、平台}

请按顺序输出：
1. 最早的有效错误，不要只看最后一行。
2. 错误类型：环境、依赖、测试数据、代码回归、flaky、权限、网络。
3. 3 个最可能假设，按概率排序。
4. 每个假设的验证命令或检查点。
5. 最小修复建议。
6. 修复后必须重跑的测试。

约束：
- 没有证据不要下确定结论。
- 如果日志里可能含密钥或用户数据，先提醒脱敏。
```

## 6. PPT Master 工作流启动模板

用途：让 Codex 先做资料分析和页级大纲，再进入 PPT Master，而不是直接生成一堆页面。

```text
我想用 Codex + PPT Master 做一份 PPT。

资料：{文件、链接或正文}
受众：{谁会看}
目标：{教学 / 汇报 / 路演 / 复盘}
页数：{大约页数}
风格：{专业 / 教学 / 学术 / 产品}

请先不要生成 PPT。

先完成：
1. 阅读资料，列出可用信息和需要核验的信息。
2. 给出 3 条叙事线，并推荐 1 条。
3. 生成页级大纲：每页标题、核心观点、证据来源、建议视觉形式。
4. 标注哪些页面信息不足。
5. 等我确认大纲后，再进入 PPT Master 生成流程。

约束：
- 不编造资料里没有的数据。
- 每页只表达一个核心观点。
- 标题尽量写成结论句。
- 导出后必须逐页验收事实、版式、讲稿和可编辑性。
```

## 7. 技术教程短视频脚本模板

用途：把技术主题写成有钩子、有证据、有画面的短视频脚本。

```text
请为 {技术主题} 写一个 {时长} 的中文技术教程短视频脚本。

目标观众：{观众}
他们的痛点：{痛点}
希望看完能做什么：{目标}
可用证据：{官方文档 / GitHub / 截图 / demo / 文章}

输出：
1. 3 个标题候选。
2. 前 10 秒钩子，必须从痛点或反差开始，不要寒暄。
3. 分镜表：时间、口播、画面动作、屏幕文字、证据来源。
4. 需要准备的素材清单。
5. 风险提醒：哪些说法需要核验，哪些不能夸大。

要求：
- 每 1-2 句口播必须有一个画面变化。
- 不写空泛口号。
- 技术步骤必须可复现。
```

## 8. AI 教程视觉提示词模板

用途：给 AI 教程做封面、知识卡片和概念图，避免“蓝色科技感大脑”这类低信息量视觉。

```text
请为这个 AI 教程主题设计视觉方案：{主题}。

目标平台：{公众号 / 小红书 / 视频封面 / 课件}
目标读者：{新手 / 开发者 / 运营 / 产品经理}
核心信息：{这张图要让读者一眼看懂什么}

请输出 4 个方向：
1. 封面钩子图。
2. 流程解释图。
3. 对比图。
4. 总结卡片。

每个方向包含：
- 画面主体
- 构图
- 关键元素
- 文字位置建议
- 配色建议
- 禁止出现的元素

禁止：
- 空泛科技感。
- 抽象 AI 大脑。
- 无信息结构的粒子背景。
- 大段正文塞进图里。
```

## 使用建议

这些模板适合放进 Codex、Claude Code、ChatGPT 或其他 Agent 工具里使用。真正使用前，先替换变量，再根据场景删掉无关约束。提示词不是越长越好，关键是目标、边界、输出格式和验收标准足够明确。
